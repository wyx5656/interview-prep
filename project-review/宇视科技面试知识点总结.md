## 一、媒体采集发送完整流程

```
采集线程 → 解析Packet → 插入SEI → RTP封装 → 队列缓存 → 转发线程发送
```

### 1. 采集
- 采集线程不断调用驱动接口 `DRV_GetVidoStream`，从驱动缓冲区取出裸码流（ES流）
- 裸码流就是直接从编码器出来的H.264/H.265码流，没有其他封装

### 2. 解析Packet
从裸码流中解析出每一帧，提取：
- 帧类型（I帧 / P帧）
- 时间戳
- 帧长度
- NALU类型

### 3. 插入SEI
SEI（Supplemental Enhancement Information）是H.264/H.265标准中的补充增强信息，可以插入自定义数据：
- 时间戳（和RTP时间戳对齐）
- 通道号
- 隐私保护状态
- 用户自定义元数据

### 4. RTP封装
给每一包加上标准RTP头（12字节），填写各个字段：
| 字段 | 说明 |
|------|------|
| **Marker (M位)** | I帧的最后一个RTP包设为1，标记帧结束 |
| **Sequence Number** | 序列号，全局连续递增，用于丢包检测 |
| **时间戳** | 采样时间戳，用于同步 |
| **Payload Type** | 标记编码格式（H.264 = 96, H.265 = 98） |
| **SSRC** | 同步源标识符，标识一个流 |

### 5. 队列缓存 转发线程发送
用一个生产者消费者队列把采集线程和转发线程解耦：

**为什么要用队列？**
1. **解耦线程**：采集只管采集封装，转发只管发送，职责分离
2. **顺序保证**：队列FIFO保证RTP包按正确顺序发送，不会乱序
3. **流量控制**：队列大小限制可以避免突发流量把网络冲爆
4. **缓解IO阻塞**：网络暂时发不出去，队列可以先缓冲，等网络恢复再发

## 二、常见问题定位排查

### 1. 客户端完全没有实况

**可能原因 & 排查步骤：**

| 可能原因 | 排查方法 |
|---------|---------|
| RTSP会话没建立成功 | Wireshark抓包看RTSP握手，哪一步失败 |
| RTP端口被防火墙屏蔽 / 绑定失败 | `netstat -anp | grep udp` 或 `ss -lup` 看端口是否正常绑定；检查防火墙/SELinux规则 |
| 客户端解析失败 | 服务端发包正常，但客户端主动发TEARDOWN → 大概率客户端不支持某些元数据格式（比如SEI自定义数据），需要客户端定位 |

**Wireshark排查步骤：**
1. 过滤 `rtsp`，看RTSP握手流程
2. 检查哪一步返回错误码（401/404/500）
3. SETUP成功后，看有没有RTP包出来

### 2. 实况花屏幕 / 马赛克

**可能原因：**
1. **RTP包顺序混乱**：序列号跳变，顺序错了客户端拼不出正确帧
2. **RTP时间戳不连续**：同步错了
3. **丢I帧 / 丢关键P帧**：没有关键帧客户端解码器刷不出新画面
4. **码率太低**：压缩太厉害，细节损失
5. **网络丢包**：随机丢包导致解码错误

**Wireshark排查：**
- `Statistics → RTP → Stream Analysis`，Wireshark会自动标红丢包和序号跳变
- 过滤 `rtp and rtp.seq > previous_rtp.seq + 1` 直接显示跳变包

### 3. 实况卡顿

**可能原因：**
| 原因 | 排查方法 |
|------|---------|
| 发送队列积压 | 看队列长度统计，如果队列一直增长说明转发跟不上采集 |
| 突发流量（I帧尖峰） | Wireshark IO Graph看短时间流量尖峰，路由器缓冲区满丢包 |
| 网络带宽不足 | Wireshark看包间隔不均匀，netstat统计UDP丢包计数上升 |
| 编码器负载过高 | 芯片编码能力不够，出帧慢 |

### 4. 码流不平滑是什么问题？怎么解决？

**原因：**
I帧比P帧大5-10倍，一帧I帧会在几十毫秒内产生几百个RTP包，如果采集线程一股脑全部塞进发送队列，就会形成**突发流量尖峰**，导致路由器缓冲区溢出丢包。

**解决方案：**
- 实现**码流平滑**（令牌桶限速），把I帧数据包打散到几百毫秒慢慢发，避免突发尖峰
- 或者I帧拆包，间隔发送，平滑网络流量

### 5. 丢帧问题

**什么情况会丢帧：**
- 发送队列满了，不得不丢帧
- 隐私保护开启后编码负载超了，编码器丢帧
- 网络带宽不足，网络丢包
- 软件bug，错误丢了I帧

**怎么抓包发现丢I帧：**
1. Wireshark过滤 `rtp`
2. `Statistics → RTP → Stream Analysis`
3. 看 `Marker` 列，Marker=1 就是I帧结束
4. 正常GOP（例如25帧）就会有一个Marker=1，如果连续上百个包都没有Marker=1，说明I帧丢了
5. 或者看RTP Payload第一个NALU类型：
   - 5 = IDR（I帧）
   - 7 = SPS
   - 8 = PPS
   - 连续很多包都没有这几个类型，就是丢了I帧

**为什么丢了I帧会马赛克：**
解码器必须收到I帧（关键帧）才能初始化参考帧，没有I帧，后续P帧都没法解码，所以会一直马赛克直到下一个I帧。

**解决方案：**
- 队列满时**优先级丢帧**：优先丢P帧，绝对保留I帧
- 动态降低帧率，给编码留出余量
- 码流平滑避免突发丢包
- 适当提高强制I帧频率，让客户端更快恢复

### 6. 发送队列积压什么原因？怎么解决？

**原因：**
- 转发线程处理慢（RTP封装耗时）
- 网络发送阻塞 `sendto()` 挂住
- 采集速度远大于发送速度

**解决：**
- 增大socket发送缓冲区 `SO_SNDBUF`
- 实现码流平滑，避免突发
- 队列满了动态丢P帧，控制队列长度
- 网络不行提醒客户端降低码率

### 7. 客户端网络不好，服务端怎么定位？

**现象：** 服务端一切正常（日志正常，发包正常），客户端还是卡/花屏。

**定位方法：**
1. **两边抓包对比**：服务端Wireshark抓包确认全发了，客户端抓包看收到多少，差很多就是网络问题
2. **看RTCP Receiver Report**：RTCP里客户端会上报丢包率、抖动，如果**丢包率高 + 抖动大**，基本就是客户端网络问题
3. ping测延迟和丢包，traceroute看哪里丢包

## 三、RTSP协商过程详解

### 基本概念
- **RTSP** 是**实时流协议**，默认用 **TCP 554端口** 做**控制信令**，只负责协商，不传输数据
- 真正的视频数据 **RTP** 用 **UDP** 传输，端口是**协商出来的**，不是固定的

### 完整协商流程

| 步骤 | 方向 | 说明 |
|------|------|------|
| **OPTIONS** | 客户端 → 服务器 | 客户端问：你支持哪些命令？ |
| **OPTIONS响应** | 服务器 → 客户端 | 服务器回复：我支持 DESCRIBE, SETUP, PLAY, TEARDOWN |
| **DESCRIBE** | 客户端 → 服务器 | 客户端要媒体描述信息 |
| **DESCRIBE响应** | 服务器 → 客户端 | 服务器返回 **SDP**，包含：编码格式、码率、分辨率、RTP端口、Payload Type |
| **SETUP** | 客户端 → 服务器 | **最关键一步**：客户端说"我要用UDP/TCP传，我的端口是多少" |
| **SETUP响应** | 服务器 → 客户端 | 服务器确认端口和传输方式，返回Transport字段 |
| **PLAY** | 客户端 → 服务器 | 客户端说：开始播放！ |
| **PLAY响应** | 服务器 → 客户端 | 服务器回复200 OK，开始发送RTP包 |
| **TEARDOWN** | 客户端 → 服务器 | 播放结束，关闭会话 |

### 常见失败原因

| 失败阶段 | 常见原因 |
|---------|---------|
| SETUP失败 | 401 Unauthorized（认证失败）、404 Not Found（URL错）、415 Unsupported Media Type（不支持编码） |
| 协商完没数据 | 防火墙挡住RTP UDP端口 → 服务端绑对端口，但中间网络不让进 |
| SDP信息不匹配 | 客户端不认Payload Type或编码格式 |

### 常见问题Q&A

**Q: RTSP协商用TCP，那RTP必须用UDP吗？**
A: 不一定，也可以RTP over TCP，就是数据和信令复用同一个TCP连接，但UDP延迟更低，直播场景一般用UDP。

**Q: UDP有协商过程吗？**
A: 没有。RTSP协商好端口，服务器直接往约定的UDP端口发包，UDP只管发，不管对方收不收得到。

**Q: 554端口是干什么的？RTP端口为什么不固定？**
A:
- 554是RTSP控制信令的固定端口，相当于"前台接待"，只谈事不干活
- 真正传数据的RTP端口是SETUP阶段动态协商出来的，服务器可以动态分配，支持多个客户端同时看不同流

## 四、核心概念澄清

### 1. Sequence Number 作用？怎么看丢包？

- **作用**：每个RTP包一个序号，**连续递增**，用来检测丢包和乱序。

- **怎么看跳变：**
  1. Wireshark选中任意RTP包 → `Statistics → RTP → Stream Analysis`
  2. 看Sequence Number列，有丢包Wireshark自动红色高亮 `Lost packets`
  3. 过滤命令：`rtp and rtp.seq > previous_rtp.seq + 1` 直接显示跳变包

- **序号跳变一定是网络丢包吗？**
  不一定：也可能是软件发包bug，序列号没有正确递增。如果服务端抓包就看到跳变，就是软件bug；服务端正常客户端收到跳变，就是网络丢包。

### 2. SPS / PPS 是什么？作用？

| 名称 | 作用 |
|------|------|
| **SPS (Sequence Parameter Set)** | 序列参数集，包含整个视频序列的全局参数：分辨率、帧率、码率、GOP结构、profile level → 解码器必须先拿到这个才能初始化解码器 |
| **PPS (Picture Parameter Set)** | 图像参数集，包含当前帧的解码参数：量化参数、熵编码方式 → 解码每一帧需要用到 |

**为什么重要：**
客户端必须先收到SPS/PPS才能正确解码后续的I帧/P帧。通常SPS/PPS放在第一个I帧的开头。如果丢了SPS/PPS，就算收到I帧也解不出来，客户端就黑屏花屏。

### 3. 什么时候需要强制I帧？

强制I帧就是让编码器立刻输出一个I帧，常见场景：
- 客户端刚连接（SETUP/PLAY成功后）→ 客户端没有参考帧，必须给一个I帧
- 分辨率 / 帧率 / 码率切换 → 参数变了，需要新的关键帧刷新
- 隐私保护开启/关闭 → 参数变化，需要刷新
- 网络丢包后，客户端通过RTCP快速恢复请求关键帧
- 录像切片开始，需要一个I帧作为切片起点

### 4. I帧瞬时数据量特别大是什么意思？为什么会有问题？

- **意思：** 一个I帧的大小通常是P帧的 **5-10倍**，所以采集到一个I帧，会在几十毫秒内产生上百个RTP包
- **问题：** 如果不做平滑，一次性全部塞进发送队列，转发线程一下子全发出去，就会在网络上形成**流量尖峰**，路由器/交换机缓冲区装不下就会丢包
- **排查：** Wireshark `Statistics → IO Graph` 能清楚看到尖峰，尖峰位置对应的就是I帧

### 5. SDP 是什么？

**SDP = Session Description Protocol**，会话描述协议，是RTSP DESCRIBE阶段服务器返回给客户端的文本描述：

包含内容：
- 视频编码格式（H.264/H.265）
- Payload Type
- RTP端口号
- 码率、帧率、分辨率
- 有时候也会带SPS/PPS

**作用：** 客户端拿到SDP才知道"这个流是什么格式，用哪个端口收，怎么解码"。

## 五、SIP协议（国标GB28181场景）

### 基本概念

- 在国标GB/T 28181场景下，**SIP代替RTSP做信令协商**，视频数据仍然用RTP传输
- SIP默认端口 **5060**（UDP/TCP），加密SIPS是5061
- RTSP多用于普通摄像头，SIP是国标强制要求，支持注册、心跳、报警、云台控制等全套能力

### 完整流程：设备注册 + 实时预览

#### 阶段一：设备注册（设备上线）
1. **REGISTER** → 设备主动注册，携带设备ID、IP、端口
2. **401 Unauthorized** → 平台要求认证
3. **REGISTER（带鉴权）** → 设备重新发送带digest认证的请求
4. **200 OK** → 注册成功
5. **心跳保活** → 每隔30秒发REGISTER心跳，平台回复200 OK

#### 阶段二：实时预览
1. **INVITE** → 平台邀请设备建立会话，请求实时预览
2. **100 Trying** → 设备收到了，正在处理
3. **200 OK + SDP** → 设备同意，返回SDP（编码、RTP端口）
4. **ACK** → 平台确认，会话建立完成
5. **RTP传输** → 设备开始往协商好的端口发RTP包
6. **BYE** → 一方结束会话

### Wireshark抓包排查方法

- 过滤器：直接输入 `sip`，想看RTP就 `sip or rtp`
- 正常流程应该看到：
  ```
  REGISTER → 401 → REGISTER（带auth） → 200 OK
  INVITE → 100 Trying → 200 OK（带SDP） → ACK → 大量RTP包
  ```
- 右键任意SIP包 → `Follow → SIP Stream` 可以一次性看完整个会话流程，非常方便

### 常见失败特征

- 返回 403 Forbidden / 404 Not Found / 486 Busy → 协商失败
- INVITE之后长时间没200 OK → 设备没响应
- 协商完成后没有RTP包 → 网络问题或端口错

## 六、OSD模块整体架构

### 概述
OSD（On-Screen Display）模块负责在视频帧上叠加文字、时间、日期、Logo、自定义内容，是软件层独立组件：
- 软件层：负责配置管理、刷新逻辑、区域行管理
- 驱动/硬件：负责实际像素渲染（调用 `DRV_` 接口）

### 能力规格
系统支持 **10个区域**，每个区域支持 **10行**，最多可叠加 10×10 = 100 个OSD元素。

有一个**独立的OSD刷新线程**，专门负责周期性更新，不影响采集和编码线程性能。

### 完整配置流程

1. 用户在Web界面添加OSD（选择类型：时间/文字/图片，配置字体、颜色、位置、透明度）
2. Web通过私有LAPI接口把配置下发给软件（JSON格式）
3. 软件接收配置后，调用 `DRV_Create()` 驱动接口创建OSD实例
4. 创建成功，软件侧记录该OSD的所有信息（ID、类型、位置、内容、属性）

### 数据结构（软件侧）
```c
// 软件维护一个OSD列表，每个OSD一个对象
struct OSD {
    int id;             // 驱动返回的OSD ID
    int type;           // 实时OSD / 静态OSD
    int region_id;      // 区域号 0-9
    int line_id;        // 行号 0-9
    char *content;      // 文字内容或图片缓冲区
    Font font;
    Color color;
    bool enabled;       // 是否启用
};
```

### 刷新线程工作逻辑

独立刷新线程循环运行：
```c
while (!stop) {
    sleep(1);  // 每秒扫描一次
    for each OSD in osd_list:
        if (!OSD.enabled) continue;
        if (OSD.type == 静态OSD):
            if (!配置变更) continue;  // 静态只有配置变更才更新
        if (OSD.type == 实时OSD):
            重新生成内容（比如当前时间）；
        if (内容变化):
            调用 DRV_Update(id, new_content);  // 更新驱动
}
```

- **静态OSD**（固定文字、Logo）：内容不变，只在配置变更时更新一次
- **实时OSD**（时间、动态信息）：内容会变，每秒更新一次

### 更新 & 销毁流程
- **更新**：内容变化，调用 `DRV_Update()` 传给驱动
- **销毁**：用户删除OSD，调用 `DRV_Destroy()` 销毁驱动实例，软件侧清除记录

### 面试1分钟口述版本（直接背）

> "我负责OSD模块的软件逻辑部分。用户在Web添加OSD配置后，通过LAPI接口下发，软件接收后调用驱动接口创建OSD实例。运行时有一个独立的刷新线程，区分实时OSD（时间日期，每秒更新）和静态OSD（固定文字图片，只在配置变化更新）。系统支持10个区域每个区域10行，刷新线程会判断内容是否变化、是否启用，只更新需要更新的，避免不必要的驱动调用，不影响采集编码性能。删除的时候调用驱动销毁接口。整个架构职责清晰，软件硬件分工明确。"

## 八、经典bug案例

### 案例一：隐私保护开启后随机性黑屏问题

#### 问题背景
隐私保护功能开启后，需要输出两路码流：
- **主码流**：原始YUV，不打码，用于录像和AI分析
- **隐私码流**：人脸打马赛克，用于实时预览

从一路变两路，总编码负载翻倍，但底层芯片编码能力是有限的。当主码流已经是高分辨率高帧率（比如2688×1520@60fps），再开一路隐私流，总负载就会超出芯片上限，编码器异常，输出不出码流，最终黑屏。

#### bug根因
前同事代码实现问题：
```c
// 原代码逻辑有bug：
static int cached_original_capability = 0;  // 静态变量保存原始编码能力

if (cached_original_capability != 0) {
    // 已经拷贝过了，不用再拷贝
} else {
    // 第一次：把全局编码能力拷贝给cached_original_capability
    cached_original_capability = global_capability;
    // 然后 global_capability /= 2;
}

// 关闭的时候：
if (cached_original_capability != 0) {
    global_capability = cached_original_capability;
}
```
**bug在哪里？** → 两个分支都判断 `cached_original_capability != 0` 才进去，第一次开启时cached就是0，所以永远进不去拷贝分支，`global_capability` 永远不会除以2！所以编码能力一直没降，负载还是超了，超出就黑屏。

#### 解决方案：动态调整编码能力

**整体策略：隐私保护开启后，总编码能力减半，不够就动态降参数，优先级 帧率 > 分辨率**

1. **开启时：**
   - 保存原始完整编码能力到全局变量备份
   - 总能力减半，计算当前配置的总负载
   - 如果总负载超出剩余能力：
     - 优先降帧率（60→30）
     - 还不够就降分辨率（2688×1520 → 2560×1440）
   - 更新能力集，影响Web界面显示（不支持的制式不显示）

2. **配置保护：**
   - 切换制式时实时校验，如果总负载超出就报错，不让配置
   - Web刷新后会拉取新能力集，不支持的不会显示

3. **Onvif处理：**
   - Onvif只在上线时拉一次能力集，所以提示用户"隐私开启后需要重新设备上线才能生效"

4. **关闭时：**
   - 恢复原始能力集，重新加载，Web和Onvif恢复原来支持的制式

#### 总结
- 这个bug是静态变量初始化判断逻辑错误，导致能力一直没降，负载超出芯片上限黑屏
- 修复后兼顾了功能和稳定性，开启隐私保护也不会黑屏了

### 案例二：四目产品从新版本回退到老版本初始化挂死

#### 问题背景
- 原代码只支持最多2通道，四目新产品出来后，最大通道改成4
- 原来人数统计智能功能只有一个通道有，四目要求四个通道都支持，所以修改了GroupID的存储逻辑
- 问题现象：新版本用一段时间，回退到老版本，双通道设备（通道2开智能）初始化阶段直接crash挂死

#### 根因分析：GroupID存储逻辑不兼容

**老版本逻辑：**
- 只有一个通道有智能，GroupID依次分配：通道1 → 0,1；通道2 → 2,3
- 配置文件只存一个有效GroupID，位置对应通道

**新版本逻辑：**
- 四个通道都可能有智能，每个通道位置都存对应的GroupID，不用就是0 0
- 所以双通道设备通道2开智能，配置文件就是 `0 0 2 3` （第一个位置0 0空着，第三个位置2 3）

**回退后老版本怎么读：**
- 老版本按旧逻辑读，直接去**第一个位置**找GroupID，读到 `0 0`，以为有效，初始化的时候用错误的GroupID找资源，越界访问 →  crash。

#### 解决方案：配置兼容

- 新增 `GroupIDNew` 字段，新版本写新字段
- 升级的时候自动做配置迁移，老数据不动，数据写到新字段
- 回退到老版本，还是读老字段，和原来一样
- 新版本可以兼容新旧两种格式，这样升级回退都不crash
