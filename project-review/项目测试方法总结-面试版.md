### 2.4 并发连接压力测试 - test_concurrent.sh

**测试目标：** 测试 epoll 边缘触发在高并发下能否处理大量并发连接。

**测试策略：**
1. 接受命令行参数指定并发连接数
2. 一次性发起 N 个并发 TCP 连接到服务器
3. 统计总耗时，计算平均每个连接建立时间
4. 检查是否失败

**怎么运行：**
```bash
# 默认 50 个并发连接
./test_concurrent.sh

# 指定 200 个并发连接
./test_concurrent.sh 200
```

**关键代码：**
```bash
START_TIME=$(date +%s%N)
for i in $(seq 1 $NUM_CLIENTS); do
    (echo -n "" | nc $SERVER_HOST $SERVER_PORT > /dev/null 2>&1) &  # 后台并行发起连接
done
wait  # 等待全部完成
END_TIME=$(date +%s%N)
ELAPSED_MS=$(( ($END_TIME - $START_TIME) / 1000000 ))
```

**测试结果示例：**

|---------|---------|---------|

| 测试2 | 10KB 随机文件上传下载 | SHA256 一致，文件完整 |
| 测试3 | 1MB 随机文件上传下载 | SHA256 一致，文件完整 |

**断点续传测试原理：**
- 客户端可以从任意 `offset` 位置开始上传/下载

- 相同内容文件第二次上传，服务端检测到SHA256已存在，直接完成，不需要传内容，节省时间

**完整性校验：** 上传前计算SHA256，下载后重新计算SHA256，对比是否一致。

### 3.6 缓存命中率测试

双缓存LRU一级缓存 + Redis二级缓存架构，测试缓存命中率：

**测试方法：**
1. 第一次访问所有查询 → 全部不命中 → 加载
2. 第二次访问相同查询 → 大部分命中一级缓存
3. 统计命中率：`命中数 / 总请求数`

**实际结果：**
- 热点查询一级缓存命中率可达 90%+
- 一级缓存miss，二级Redis命中率可达 95%+
- 只有不到 5% 请求需要走到后端计算

**性能收益：**
- 一级缓存命中：平均延迟 ~1-2ms
- Redis命中：平均延迟 ~5-10ms
- 后端计算：平均延迟 ~20-50ms
- 高命中率大大降低平均延迟

## 四、性能指标含义说明

| 指标 | 含义 | 单位 | 越大越好还是越小越好 |
|------|------|------|---------------------|
| **QPS (Queries Per Second)** | 每秒能处理多少个查询 | req/s | 越大越好 |
| **延迟 (Latency)** | 从请求发出去到收到响应的时间 | ms (毫秒) | 越小越好 |
| **平均延迟** | 所有请求延迟的算术平均 | ms | 越小越好 |
| **中位数延迟 (p50)** | 一半请求比这个快，一半比这个慢 | ms | 越小越好，比平均更能代表典型情况 |
| **P95延迟** | 95%的请求延迟比这个小，剩下5%更慢 | ms | 越小越好，衡量长尾延迟 |
| **P99延迟** | 99%的请求延迟比这个小 | ms | 越小越好，衡量最差情况 |
| **错误率** | 失败请求占总请求的百分比 | % | 越小越好 |
| **命中率** | 缓存命中请求占总请求比例 | % | 越大越好 |
| **并发连接数** | 同时保持的连接数 | 个 | 能支持越多越好，说明架构扩展性好 |
| **吞吐量** | 单位时间能处理的数据量 | MB/s | 越大越好 |

## 六、面试官可能问到的测试相关问题

### Q1: 你们项目怎么做测试的？

**A：**
> 我的两个项目都分层测试：
> 1. **单元测试**：核心模块比如JWT、最小编辑距离、双缓存LRU都有单独的单元测试，保证每个模块独立正确
> 2. **集成测试**：启动整个服务，测试端到端功能
> 3. **功能测试**：Cloudisk测试上传下载断点续传，搜索引擎测试搜索和搜索推荐API
> 4. **性能测试**：测量延迟、QPS，验证优化效果
> 5. **并发压力测试**：用高并发跑，验证稳定性和并发安全

### Q2: 什么是P95延迟？为什么要看P95？

**A：**
> P95延迟就是把所有请求的延迟排序，第95百分位那个延迟值。意思是95%的请求延迟都比它小，只有5%的请求更慢。
>
> 为什么要看P95：平均延迟容易被少数快请求掩盖，长尾延迟高用户体验还是不好。比如100个请求99个1ms，1个1000ms，平均才11ms看起来不错，但有一个用户1000ms体验很差。P95能反映出长尾问题，更能体现服务质量。

### Q3: 怎么测试并发安全？怎么定位竞态条件？

**A：**
> 测试方法：
> 1. 写压力测试脚本，用很多线程/协程同时访问共享资源
> 2. 跑很多次，看会不会出现数据损坏或者崩溃
> 3. 用工具帮助：比如ThreadSanitizer可以检测竞态条件
>
> 定位竞态条件：
> 1. 先确定哪些是共享资源
> 2. 检查每个共享资源访问有没有正确加锁
> 3: 加日志看访问顺序，是不是出现了不一致

### Q4: 性能测试和功能测试区别是什么？

**A：**
> - **功能测试**：验证功能对不对，是不是符合需求，主要关注正确性
> - **性能测试**：验证快慢，测量延迟、QPS、吞吐量，主要关注性能指标

### Q5: 缓存命中率怎么测试计算？

**A：**
> 缓存命中率 = 命中次数 / 总访问次数。测试方法：
> 1. 模拟真实访问流量，访问顺序基本符合幂律分布（少数热点被访问多次）
> 2. 每次访问记录是命中还是缺失
> 3. 最后统计比例。我的搜索引擎项目，热点查询一级缓存命中率90%+，大部分查询内存直接返回，所以延迟很低。

### Q6: 你做项目过程中发现过什么Bug？怎么测试复现修复验证？

**可以说云盘项目：**
> 我印象最深的是下载断点续传的偏移量bug，原来代码错误地把偏移量按内存页4096对齐，导致下载下来文件内容偏移错位，SHA256不对。
>
> 修复过程：
> 1. 测试：我写了Python自动化测试脚本，从不同偏移位置开始下载，下载完对比SHA256
> 2. 定位：发现对齐是多余的，去掉对齐直接用实际偏移就行
> 3. 验证：修复后再跑测试脚本，各种偏移都能得到正确SHA256，问题解决
>
> 还有哈希表冲突覆盖节点bug，原来插入冲突节点直接覆盖了，我改成链表插入，测试就是插入多个肯定冲突的key，都能正确找到，修复了。

### Q7: 压力测试有什么意义？

**A：**
> 1. **验证稳定性**：高并发下会不会崩溃，有没有内存泄漏，错误率是不是0
> 2. **找出瓶颈**：测出来最大QPS能到多少，延迟在并发下会不会飙升
> 3. **验证优化效果**：优化前后对比，看看优化真的提升了性能吗
> 我的搜索引擎在16并发1000请求下，错误率0，QPS能到377，说明稳定。
