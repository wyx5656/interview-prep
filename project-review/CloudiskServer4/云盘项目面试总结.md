# 云盘项目面试总结

---

## 1. 项目介绍

**CloudiskServer** 是一个基于 C 语言实现的**多人网络云盘服务器**，支持多用户登录注册，用户可以在云端进行文件管理（pwd/cd/ls/mkdir/rmdir），支持大文件上传下载，并且支持**断点续传**和**秒传**功能。

我对项目进行了全面的性能优化和架构重构，主要做了这几件事：

1. **IO 模型优化**：将 epoll 从水平触发改为**边缘触发 + 非阻塞 Socket**，提升高并发性能
2. **数据库连接池优化**：从固定大小改为**支持动态扩容**，应对突发流量
3. **功能增强**：修复下载断点续传偏移量计算 bug，给上传也添加了**断点续传支持**
4. **架构重构**：按照最佳实践重构为 **"主线程处理短任务，线程池处理长任务"** 的简洁架构，删除了原来的双端口设计，代码更简洁，性能更好
5. **代码审查**：修复了多个并发安全和内存安全问题（double free、竞态条件、内存泄漏等）

## 2. 整体架构设计

### 架构图

```
┌─────────────────────────────────────────────────────────────┐
│                    主线程                                    │
│                    │
│                  epoll_wait
│                    │
│                    ▼
│              读取解析命令
│                    │
│          ┌─────────┴─────────┐
│          │                 │
│  短任务(pwd/ls/cd/...)   长任务(puts/gets上传下载)
│  直接在主线程执行         入队到长任务线程池
│          │                 │
│          ▼                 ▼
│        返回响应       线程池工作线程异步执行
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
                        数据库连接池（动态扩容）
```

### 为什么这么设计？

| 设计决策 | 原因 |
|---------|------|
| **epoll 而不是 select/poll** | select 有连接数限制（默认 1024），时间复杂度 O(n)；epoll 时间复杂度 O(1)，性能更好 |
| **边缘触发而不是水平触发** | 水平触发在缓冲区还有数据时会一直唤醒，边缘触发只在状态变化时唤醒一次，减少不必要的唤醒，减少上下文切换，高并发下性能更好 |
| **短任务主线程直接执行** | 短任务很快就能完成（几ms），不需要进线程池排队，直接执行延迟最低 |
| **长任务线程池执行** | 上传下载是慢IO，交给后台线程池执行，主线程可以继续处理新请求，不会被阻塞 |
| **单端口而不是双端口** | 原来开两个端口（一个控制，一个传输）完全没必要，客户端本身用多线程已经能做到一个连接上同时传文件和输命令，单端口更简洁 |

## 3. 核心模块实现

### (1) epoll 边缘触发

**关键点：**
- 必须把 socket 设置为 **非阻塞**（`fcntl` 设置 `O_NONBLOCK`）
- 需要循环 `recv` 直到返回 `-1` 且 `errno == EAGAIN/EWOULDBLOCK`，表示数据已经读完

**面试官常问：水平触发和边缘触发的区别？**
> - LT：只要缓冲区有数据就会一直唤醒，编程简单，但效率低
> - ET：只有状态变化（从无到有）时唤醒一次，效率高，但需要循环读到 EAGAIN，编程稍微复杂一点
> - 高并发下 ET 比 LT 性能好，减少不必要的唤醒

---

### (2) 数据库连接池动态扩容

**设计：**
- **初始连接数**：从配置读取，比如 3
- **最大连接数**：从配置读取，比如 20
- **动态扩容策略**：当所有连接都被占用，且当前连接数 < 最大连接数时，**自动新建一个连接**加入池子，应对突发流量
- 达到最大连接数后，新来的请求阻塞等待，保护数据库不被压垮

**为什么要动态扩容？**
> 正常情况下大部分时间连接数用不到那么多，一开始只开少数连接节省资源，遇到突发流量再自动扩容，兼顾资源效率和应对突发。

---

### (3) 断点续传实现

**协议设计：**

**上传：**
1. 客户端发送文件名 + SHA256 + 文件长度
2. 客户端检查本地已上传多少，发送 `offset` 给服务端
3. 服务端：
   - 如果 SHA256 已经存在全局文件表 → **秒传**，直接创建虚拟文件即可
   - 如果不存在 → 从 `offset` 位置开始接收数据，`mmap` 映射文件从偏移位置开始写
4. 接收完成后插入数据库记录

**下载：**
1. 客户端请求下载文件
2. 服务端返回文件长度
3. 客户端发送已下载 `offset` 给服务端
4. 服务端从 `offset` 开始发送数据

**修复了什么 bug？**
> 原来下载偏移量错误地按 4096 对齐，导致文件损坏，修复后直接使用实际文件大小作为偏移，完全准确。

---

### (4) 超时管理（时间轮实现）

**需求：** 长时间不操作的连接自动断开，节省资源。

**实现：** 使用**时间轮算法**
- 数组大小 `TIMEOUT = 30`，对应 30 秒超时
- 每个槽位存放该时间点到期的用户链表
- 每秒移动一次指针，清空当前槽位，断开所有到期连接
- 用哈希表（链表解决冲突）存储 `sockfd → 时间槽索引` 映射，快速查找

**为什么用时间轮？**
> 到期时间都是整秒，移动一次处理一批，效率很高，实现简单。

---

### (5) 并发安全

**问题：** 主线程插入删除用户，工作线程查找用户，存在并发访问。

**解决方案：** 给全局哈希表 `userTable` 加 `pthread_mutex_t` 互斥锁，所有查找、插入、删除都需要拿锁。

## 4. 项目亮点（面试时说）

1. **性能优化到位**：epoll 边缘触发，短任务主线程直接执行，平均延迟 **0.61ms**，吞吐量 **1150 QPS**，高并发下性能很好

2. **架构简洁优雅**：删除了不必要的双端口设计，单端口长短任务分离，代码更简洁，结构更清晰

3. **功能完整**：支持断点续传（上传下载都支持）、秒传（基于 SHA256）、用户登录注册 JWT 认证

4. **处理了很多边界情况**：修复了 double free、竞态条件、哈希碰撞覆盖、内存泄漏等问题，代码健壮性好

5. **数据库连接池动态扩容**：兼顾了资源利用率和应对突发流量的能力

## 5. 你在这个项目中遇到了什么难点？是怎么解决的？

**示例回答：**

> 我觉得最大的收获是架构重构的过程。原来项目是双端口，一个端口处理控制，一个端口处理传输，我觉得设计冗余，讨论后改成单端口长短任务分离，主线程处理短任务保证低延迟，长任务交给线程池，重构之后性能反而更好，代码也更少。这个过程让我对"怎么设计更简洁高效的架构"有了更深的理解。

> 另外一个就是代码审查的时候，发现了几个并发问题，比如哈希表没有锁，超时队列哈希冲突会覆盖节点，这些问题平时功能测试不容易发现，但高并发下会出问题，修复之后系统更健壮了。

## 6. 面试官可能追问

### Q: 为什么不用多个线程池，一个处理短任务一个处理长任务？
**A:** 短任务本身很快，主线程直接执行已经能得到最低延迟，交给线程池反而需要排队，增加延迟。而且我们只有长任务需要线程池，一个就够了，设计更简单。

### Q: epoll 边缘触发会不会丢失数据？
**A:** 不会，只要循环读到 EAGAIN，就能把所有数据读完，不会丢。我就是这么实现的。

### Q: 秒传原理是什么？
**A:** 上传文件时先算 SHA256，如果服务端已经有这个 SHA256 的文件，说明内容相同，直接复用已存储的文件，在用户目录下创建一个引用就行，不需要重复存储，节省空间，上传也秒完成。

### Q: 什么是惊群效应？你的项目里怎么处理的？
**A:** 惊群就是当多个线程阻塞在 accept 上，一个连接进来所有线程都被唤醒。这里我项目只有一个主线程做 epoll，所以没有惊群问题，长任务都在线程池，不会有这个问题。

### Q: 为什么用 mmap 传输文件？
**A:** mmap 可以省去用户态到内核态的数据拷贝，直接操作文件内存，简化代码，性能也更好。sendfile 也可以零拷贝，mmap 配合断点续传从偏移开始也很方便。

### Q: LRU 你知道吗？超时淘汰为什么不用 LRU？
**A:** LRU 是淘汰最近最少使用，这里需求很简单就是 30 秒超时自动断开，时间轮正好满足，实现简单，效率也够，不需要 LRU。我自己也实现过 LRU（就是双向链表加哈希表，get 和 put 都是 O(1)），但这个场景不需要。
