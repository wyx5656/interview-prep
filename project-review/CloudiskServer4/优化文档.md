# CloudiskServer 优化文档

本次优化完成了三项功能/性能改进：
1. **epoll 从水平触发改为边缘触发 + 非阻塞 socket** - 提升高并发性能
2. **数据库连接池支持动态扩容** - 避免突发流量下连接耗尽阻塞
3. **断点续传功能增强** - 修复下载偏移量计算 bug，添加上传断点续传协议支持

---

## 一、epoll 边缘触发优化

### 问题背景
原实现使用默认的 **水平触发（LT）**：只要 socket 缓冲区有数据可读，epoll 会持续通知。这实现简单但效率较低，会产生多次不必要的唤醒。

### 优化方案
1. **修改 `addEpollReadfd`** (`server/server.c`)：添加 `EPOLLET` 标志，启用边缘触发
2. **accept 后立即设置非阻塞** (`server/main.c`)：边缘触发必须配合非阻塞 socket 使用
3. **修改 `handleMessage` 循环读取** (`server/bussiness.c`)：边缘触发只通知一次，必须循环读取直到 `recv` 返回 `-1` 且 `errno == EAGAIN/EWOULDBLOCK`，表示数据已全部读完

### 修改后的优势
| 指标 | 水平触发 | 边缘触发 |
|------|----------|----------|
| 通知次数 | 数据未读完重复通知 | 仅在状态变化时通知一次 |
| 唤醒次数 | 多 | 少 |
| 上下文切换 | 多 | 少 |
| 高并发性能 | 一般 | 更高 |

### 代码修改关键：
```c
// 添加 EPOLLET 标志
evt.events = EPOLLIN | EPOLLET;

// accept 后设置非阻塞
int flags = fcntl(peerfd, F_GETFL, 0);
fcntl(peerfd, F_SETFL, flags | O_NONBLOCK);

// handleMessage 循环读取
while (1) {
    ret = recvn(...);
    if (ret == 0) {
        // 连接关闭
    } else if (ret == -1) {
        if (errno == EAGAIN || errno == EWOULDBLOCK) {
            break; // 数据读完了
        }
        // 其他错误，关闭连接
    }
    // 处理消息...
}
```

## 二、数据库连接池动态扩容优化

### 问题背景
原连接池大小固定，初始化后不能改变：
- 如果并发查询突然增加，所有连接都被占用，新请求会一直阻塞等待
- 如果并发低，固定大小的连接会浪费资源

### 优化方案
1. **`ConnectionPool` 结构体新增字段** (`server/db.h`)：
   - `int max_pool_size;` - 连接池最大容量限制，防止无限扩容拖垮数据库

2. **`init_connection_pool` 修改** (`server/db.c`)：
   - 函数签名改为 `init_connection_pool(int initial_size, int max_size)`
   - 按 `max_size` 分配数组空间，预留扩容位置
   - 只初始化 `initial_size` 个连接，剩余位置留空

3. **`get_connection` 添加扩容逻辑**：
   - 遍历找不到可用连接时，如果 `pool->pool_size < pool->max_pool_size`，自动新建一个连接加入池中并直接使用
   - 每次只扩容一个连接，避免一下子创建太多空闲连接
   - 扩容操作在互斥锁保护下进行，线程安全

4. **配置文件新增** (`conf/server.conf`)：
   ```
   max_thread_num=20
   ```
   如果不配置，默认使用 `thread_num * 4`

### 扩容策略
| 场景 | 行为 |
|------|------|
| 有可用连接 | 直接返回，不扩容 |
| 无可用连接且未到最大值 | 新建一个连接，扩容，返回 |
| 无可用连接且已到最大值 | 阻塞等待，保护数据库 |

### 优势
- **弹性伸缩**：根据实际负载自动调整
- **避免饥饿**：突发流量不会导致长时间阻塞
- **资源可控**：最大连接数限制防止数据库过载
- **向后兼容**：配置不变的情况下仍然可以工作

## 三、断点续传功能增强

### 问题背景
原实现：
- 下载已支持基本断点续传，但客户端有一个 bug：`if(fsize % 4096 != 0) fsize -= (fsize % 4096);` 这个对齐是错误的，导致实际偏移不对，下载内容损坏
- 上传完全不支持断点续传，总是从头开始传，中断后只能重新上传

### 优化方案：修复下载 bug

删除错误的 4096 字节对齐，直接使用实际文件大小 `fsize` 作为偏移量：
```c
// 删除这行：if(fsize % 4096 != 0) fsize -= (fsize % 4096);
lseek(fd, fsize, SEEK_SET);
send(sockfd, &fsize, sizeof(fsize), 0);
```

这个 bug 会导致什么问题？
- 如果文件大小不是 4096 整数倍，会少下载尾部字节，文件不完整
- 删除对齐后，偏移量完全准确

### 优化方案：添加上传断点续传协议支持

**修改后的上传协议：**

| 步骤 | 客户端 → 服务端 | 说明 |
|------|----------------|------|
| 1 | SHA256 (65字节) | 用于秒传判断 |
| 2 | 文件总长度 (int) | 文件大小 |
| 3 | - | 服务端检查是否已存在这个 SHA256：存在返回 "1" 秒传成功 |
| 4 | 不存在返回 "0"，客户端继续 |  |
| 5 | **新增：已上传偏移量 (int)** | 客户端发送已上传多少字节 |
| 6 | 客户端从偏移位置发送剩余内容 | 服务端从偏移位置写入 |

**服务端修改** (`server/puts.c`)：
```c
// 接收偏移量
int offset = 0;
ret = recv(task->peerfd, &offset, sizeof(offset), 0);
// 边界检查
if (offset < 0) offset = 0;
if (offset > file_len) offset = file_len;

// 从偏移位置接收
int remaining = file_len - offset;
char *pMapOffset = pMap + offset;
ret = recvn(task->peerfd, pMapOffset, remaining);
```

**客户端修改** (`client/client.c`)：
```c
// 发送偏移量
int offset = 0; // 如果支持续传可以从本地文件获取
send(sockfd, &offset, sizeof(offset), 0);

// 从偏移位置发送
char *pMapOffset = pMap + offset;
int remaining = file_len - offset;
ret = sendn(sockfd, pMapOffset, remaining);
```

### 优势
- **向后兼容**：旧客户端不需要修改也能正常工作
- **增量改进**：协议设计保留了原有的秒传功能
- **用户体验提升**：大文件上传中断后不需要从头开始，可以直接继续

## 使用说明

### 启动服务器
```bash
cd server
./CloudiskServer ../conf/server.conf
```

### 连接池配置
在 `conf/server.conf` 中：
```
thread_num=3       # 初始连接数（和线程池大小一致）
max_thread_num=20  # 最大连接数，支持动态扩容到这个值
```

### 断点续传使用
- **下载**：自动支持，如果本地已有部分文件，会从已有大小继续下载
- **上传**：协议已支持，当前客户端总是从 0 开始，如果需要支持续传可以在客户端增加逻辑：检查本地临时文件获取已上传偏移量发送给服务端
