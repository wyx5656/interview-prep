## 一、项目整体介绍

### 1.1 项目做了什么

这是一个基于 **C语言 Linux 网络编程** 实现的**多用户并发网络云盘存储服务器**，类似百度网盘的后端服务。支持：

- **用户注册登录**：密码加盐加密存储，JWT Token 认证
- **虚拟文件系统**：每个用户拥有独立的目录空间，支持 `pwd/cd/ls/mkdir/rmdir` 命令
- **文件上传下载**：支持大文件传输，**断点续传**（上传下载都支持）
- **秒传功能**：基于 SHA256 文件内容哈希，相同文件只存储一份，节省空间秒级完成
- **零拷贝传输**：使用 `mmap` 映射文件，减少内存拷贝，提升传输性能
- **超时自动断开**：使用**时间轮算法**管理空闲连接，30秒无操作自动断开节省资源

### 1.2 技术栈

| 模块 | 技术 | 说明 |
|------|------|------|
| 语言 | C11 | 系统级编程，直接控制内存 |
| 网络 | POSIX Socket + epoll边缘触发 | I/O多路复用，高并发性能好 |
| 并发 | pthread线程池 + 任务队列 | 生产者消费者模型，长短任务分离 |
| 数据库 | MySQL C API + 连接池 | 存储用户信息和文件元数据 |
| 认证 | crypt()加盐加密 + JWT | 密码安全存储，无状态认证 |
| 文件 | mmap零拷贝 + SHA256 | 高效传输，秒传去重 |
| 超时管理 | 时间轮算法 | O(1)批量处理到期连接 |

## 二、项目整体架构设计

```
┌─────────────────────────────────────────────────────────────┐
│                    主线程 (epoll 事件循环)                   │
│                                                             │
│  epoll_wait() 等待事件 → 读取数据 → 解析命令                 │
│         │                                                   │
│   ┌─────┴─────┐                               ┌─────────────┘
│   │ 短任务    │  （pwd/ls/cd/mkdir/rmdir）    │ 长任务      │
│   │ 直接执行  │                               │ （puts/gets上传下载） │
│   └─────┬─────┘                               └─────┬───────┘
│         │                                           │
│         ▼                                           ▼
│      返回响应                             入队到线程池任务队列
│                                               │
│                                               ▼
│                                        线程池工作线程异步执行
│                                               │
│                                               ▼
│                                        MySQL连接池（动态扩容）
│                                               │
│                                               ▼
│                                        操作数据库 / 传输文件
└─────────────────────────────────────────────────────────────┘
```

### 为什么这么设计？

| 设计决策 | 原因 |
|---------|------|
| **epoll 边缘触发 ET** | 相比水平触发 LT，减少不必要的唤醒，减少系统调用，高并发性能更好 |
| **非阻塞 Socket** | 边缘触发必须配合非阻塞，循环读直到 EAGAIN 保证读完所有数据 |
| **长短任务分离** | 短任务（pwd/ls/cd）几ms就能完成，主线程直接执行延迟最低；长任务（上传下载）慢IO交给线程池不阻塞主线程 |
| **单端口设计** | 删除原来不必要的双端口设计（一个控制端口一个传输端口），架构更简洁，客户端多线程天然能在一个连接上同时处理命令和传输 |
| **动态连接池** | 初始连接少节省资源，突发流量自动扩容，达到上限后排队保护数据库，兼顾资源效率和稳定性 |

### 一次完整请求处理流程

1. 客户端建立TCP连接 → 注册到epoll
2. 客户端发送命令 → epoll通知可读 → 主线程循环读取直到EAGAIN → 解析命令
3. 如果是短命令 → 主线程直接执行 → 写回响应
4. 如果是长命令（上传/下载）→ 封装成任务加入任务队列 → 线程池工作线程取出执行 → 执行完成写回响应

## 三、核心模块详解

### 3.1 epoll 边缘触发 vs 水平触发

**代码位置：** `server/main.c` epoll 初始化和事件循环

**关键点：**
```c
// 设置socket为非阻塞
fcntl(sockfd, F_SETFL, O_NONBLOCK);

// epoll_event 使用 EPOLLET
ev.events = EPOLLIN | EPOLLET;
```

| 对比 | 水平触发 LT | 边缘触发 ET |
|------|------------|-------------|
| 通知时机 | 只要缓冲区有数据就一直通知 | 只有从无变有（新数据到来）才通知一次 |
| 系统调用 | 多次 | 一次 |
| 性能 | 较低 | 较高 |
| 要求 | 可以阻塞读 | 必须非阻塞读，循环读到EAGAIN |

**必须循环读的原因：** 边缘触发只通知一次，如果一次没读完剩下的数据就不会再通知，所以必须循环读直到 `recv() 返回 -1 且 errno == EAGAIN/EWOULDBLOCK` 才停止。

### 3.2 线程池与任务队列

**代码位置：** `server/thread_pool.c/h` `server/task_queue.c/h`

**生产者消费者模型：**
```c
// 生产者（主线程）
void add_task(task_queue_t *queue, task_t task) {
    pthread_mutex_lock(&queue->mutex);
    queue->tasks[queue->tail] = task;
    queue->tail = (queue->tail + 1) % QUEUE_SIZE;
    pthread_mutex_unlock(&queue->mutex);
    pthread_cond_signal(&queue->cond);
}

// 消费者（工作线程）
task_t get_task(task_queue_t *queue) {
    pthread_mutex_lock(&queue->mutex);
    while (is_empty(queue)) {
        pthread_cond_wait(&queue->cond, &queue->mutex);
    }
    task_t task = queue->tasks[queue->head];
    queue->head = (queue->head + 1) % QUEUE_SIZE;
    pthread_mutex_unlock(&queue->mutex);
    return task;
}
```

**为什么用 while 循环等待？** → 处理**虚假唤醒**：多个线程同时被唤醒，一个拿走了任务，另一个醒来队列又空了，所以必须继续等。

### 3.3 动态扩容数据库连接池

**代码位置：** `server/db.c/h`

**设计：**
- **初始连接数**：从配置文件读取，默认 3
- **最大连接数**：从配置文件读取，默认 20
- **空闲链表**：保存当前空闲连接
- **并发安全**：用互斥锁保护空闲链表

**动态扩容策略：**
1. 用户请求连接 → 空闲链表有连接 → 取出返回
2. 空闲链表空 → 如果当前总连接数 < 最大连接数 → 新建一个连接返回
3. 已经达到最大连接数 → 等待条件变量，有连接放回后被唤醒再取出

**为什么要动态扩容？**
- 正常流量下，连接数用不了多少，初始少量连接节省数据库资源
- 遇到突发并发流量，所有连接都被占了，自动扩容应对突发
- 设置最大连接数，防止雪崩，保护数据库不会被太多连接压垮

### 3.4 断点续传实现

**上传断点续传流程：**

```
1. 客户端：选择要上传文件 → 计算文件 SHA25 → 获取文件大小
2. 客户端：查询本地已经上传了多少字节 → 得到 offset
3. 客户端：发送 文件名 + SHA256 + 文件大小 + offset 给服务端
4. 服务端：
   - 如果 SHA256 已经存在 global_file 表 → 秒传！直接在用户目录创建虚拟文件 → 完成
   - 如果不存在 → 打开本地存储文件，从 offset 位置开始用 mmap 映射 → 接收数据写入
5. 传输完成 → 插入 virtual_file 记录 → 返回成功
```

**下载断点续传流程：**

```
1. 客户端：请求下载某个文件 → 发送已经下载了多少字节 offset
2. 服务端：根据文件名查到文件信息 → 返回 文件总大小 + SHA256
3. 服务端：从 offset 位置开始，mmap 映射从 offset 开始发送数据
4. 客户端：接收数据，写入本地文件从 offset 位置开始
```

**修复了什么Bug？**
- 原来下载偏移量错误地按内存页4096对齐，导致文件内容偏移错误 → 修复为直接使用实际偏移，完全准确。

### 3.5 秒传（文件去重）原理

**核心思想：** 相同内容的文件只存储一份，节省服务器空间，上传瞬间完成。

**实现步骤：**
1. 客户端对要上传文件计算 SHA256 哈希
2. 把 SHA256 发给服务端
3. 服务端查询 `global_file` 表，如果已经有这个 SHA256 → 直接在用户虚拟目录创建一个指向这个文件的记录 → 完成！不需要传文件内容
4. 如果没有 → 需要正常上传，上传完成后把 SHA256 存入 `global_file`

**为什么 SHA256 可靠？** → 不同内容碰撞概率几乎为0，可以认为相同SHA256就是相同文件。

### 3.6 超时队列（时间轮算法）

**代码位置：** `server/timeout_queue.c/h`

**需求：** 空闲连接超过30秒自动断开，节省资源。

**时间轮设计：**

```
数组大小 = TIMEOUT = 30 （30秒超时）
每个数组元素是一个链表，保存该槽位到期的连接

每秒钟：
  1. 指针移动到下一个槽位
  2. 遍历当前槽位所有连接，全部断开
  3. 清空当前槽位

每个连接：
  有操作时 → 从原来槽位删除，放到新的到期槽位
```

**哈希表加速查找：** `hashtable` 存 `sockfd → 链表节点指针`，O(1)快速找到连接节点，从原来槽位删除。

**为什么用时间轮？**
- 需求简单：固定超时时间30秒，时间轮正好匹配
- 实现简单，效率高：每秒只处理一个槽位，O(1)批量处理
- 这个场景不需要 LRU，时间轮足够用，更简单高效

### 3.7 mmap 零拷贝传输

**为什么用 mmap？**
- 传统 `read()` 方式：文件 → 内核 → 用户 → 内核 → 网络，两次拷贝
- mmap 直接把文件映射到用户空间地址，只需要一次拷贝到内核，减少一次拷贝
- 配合断点续传从偏移开始使用也很方便

**代码示例：**
```c
int fd = open(file_path, O_RDWR);
char *map = mmap(NULL, file_size, PROT_READ | PROTAC_WRITE, MAP_SHARED, fd, offset);
// 直接操作map就是操作文件，不需要read/write，写完自动同步
munmap(map, file_size);
close(fd);
```

### 3.8 JWT 认证

**代码位置：** `server/jwt.c/h`

**流程：**
1. 用户登录成功 → 服务端签发 JWT Token 返回给客户端
2. 客户端后续每次请求都带上 Token
3. 服务端验证 Token 有效才处理请求

**优点：** 无状态，不需要服务端存储会话信息，扩展方便。

### 3.9 哈希表

**代码位置：** `server/hashtable.c/h`

**项目中用途：**
- 全局用户表：`sockfd → user_info` 映射，O(1)查找
- 超时队列：`sockfd → 链表节点` 映射，O(1)查找

**实现：** 数组 + 链表法解决冲突。

**修复的Bug：** 原来插入新节点时，处理哈希碰撞会覆盖前一个节点 → 修复为正确插入链表头，不覆盖。

## 四、数据库设计

### 4.1 三张核心表

```sql
-- 1. 用户表
CREATE TABLE user (
    id INT PRIMARY KEY AUTO_INCREMENT,
    username VARCHAR(50) UNIQUE NOT NULL,
    password VARCHAR(255) NOT NULL,  -- 加盐加密后的密码
    root_path VARCHAR(255) NOT NULL  -- 用户根目录路径
);

-- 2. 全局文件表（去重用）
CREATE TABLE global_file (
    id INT PRIMARY KEY AUTO_INCREMENT,
    sha256 VARCHAR(64) UNIQUE NOT NULL,  -- 文件SHA256哈希
    size BIGINT NOT NULL,                -- 文件大小
    ref_count INT NOT NULL              -- 被多少用户引用，计数为0可以删除
);

-- 3. 虚拟文件表（每个用户的目录结构）
CREATE TABLE virtual_file (
    id INT PRIMARY KEY AUTO_INCREMENT,
    user_id INT NOT NULL,               -- 属于哪个用户
    parent_id INT NOT NULL,             -- 父目录ID
    filename VARCHAR(255) NOT NULL,     -- 文件名
    is_dir BOOLEAN NOT NULL,            -- 是否是目录
    global_file_id INT,                 -- 关联到global_file
    size BIGINT,                        -- 文件大小
    create_time DATETIME NOT NULL,
    FOREIGN KEY (user_id) REFERENCES user(id),
    FOREIGN KEY (global_file_id) REFERENCES global_file(id)
);
```

### 4.2 设计思路

- **global_file 是内容层**：按内容去重，相同内容只存一份，节省磁盘
- **virtual_file 是用户视图层**：每个用户看到自己的目录树，同一个内容可以被多个用户引用
- **引用计数**：删除文件时引用计数减1，计数为0才真正删除物理文件，安全节省空间

## 五、并发编程与线程安全

### 5.1 哪些共享资源需要保护？

| 共享资源 | 保护方式 |
|---------|----------|
| 任务队列 | `pthread_mutex_t` 互斥锁 + `pthread_cond_t` 条件变量 |
| 数据库连接池空闲链表 | 互斥锁保护 |
| 全局用户哈希表 | 主线程插入删除，工作线程查找，互斥锁保护 |
| 超时队列哈希表 | 互斥锁保护 |

### 5.2 数据库连接池并发安全

```c
// 获取连接
pthread_mutex_lock(&pool->mutex);
while (pool->free_list == NULL && pool->total_connections >= pool->max_connections) {
    pthread_cond_wait(&pool->cond, &pool->mutex);
}
// ... 获取连接
pthread_mutex_unlock(&pool->mutex);

// 放回连接
pthread_mutex_lock(&pool->mutex);
// ... 放回空闲链表
pthread_cond_signal(&pool->cond);
pthread_mutex_unlock(&pool->mutex);
```

### 5.3 死锁避免

项目中没有多个锁嵌套，只有单锁情况，不会死锁。所有地方都是只需要一个锁，用完立即释放，不会长时间持有。

## 六、性能优化总结

### 6.1 优化项对比

| 优化 | 优化前 | 优化后 | 提升 |
|------|--------|--------|------|
| epoll 水平触发改边缘触发 | 多次系统调用 | 一次系统调用 | 高并发下减少上下文切换，QPS提升约15% |
| 双端口改单端口长短分离 | 架构冗余，两个端口管理复杂 | 架构简洁，主线程短任务直接执行 | 代码减少约20%，短任务延迟降低 |
| 连接池静态改动态扩容 | 固定大小，突发流量处理不好 | 初始节省资源，突发自动扩容 | 资源利用率提升，应对突发能力增强 |
| 用户查找从链表遍历改哈希表 | O(n)查找 | O(1)查找 | 1000并发用户，查找速度提升约50倍 |
| mmap代替read/write | 两次拷贝 | 一次拷贝 | 大文件传输性能提升约20-30% |

### 6.2 性能测试结果

测试环境：本地虚拟机，8核CPU

| 指标 | 数值 |
|------|------|
| 平均延迟 | 0.61ms |
| QPS | 1150 |

短命令（pwd/ls/cd）主线程直接执行，几乎就是网络延迟，非常快。

## 七、C语言知识点详解

### 7.1 指针和数组

**指针和数组区别：**
- 数组：数组名是数组首地址，在栈上（局部数组）或数据段（全局数组），大小固定
- 指针：指针变量本身占8字节（64位），存储地址，可以指向任意地址

项目中到处都用指针，链表、哈希表都靠指针实现。

### 7.2 结构体与链表

**链表节点定义：**
```c
typedef struct node_s {
    void *data;
    struct node_s *next;
} node_t;
```

**插入删除操作需要改指针，所以要传二级指针。**

### 7.3 内存管理 malloc/free

项目中所有动态内存都是 `malloc` 分配，`free` 释放，修复了多处内存泄漏。

**内存泄漏常见原因：**
- 申请了没释放
- 异常路径没处理，提前返回没free
- double free：释放之后指针没置空，重复释放 → 项目中修复了这个问题

### 7.4 静态变量 static

**static三种用法：**
1. **全局变量前static**：作用域限制在本文件，其他文件不能访问 → 项目中静态内部函数都用static
2. **函数内static**：变量存储在静态存储区，函数调用结束不销毁，保留值
3. **函数前static**：函数只能在本文件调用，其他文件不能访问

### 7.5 宏定义 #define

**项目中常用：**
```c
// 条件检查，出错跳转清理
CHECK(point, errstr, goto_label) do { \
    if (!(point)) { \
        log_err(errstr); \
        goto goto_label; \
    } \
} while(0)
```

**为什么用 do {} while(0)？** → 宏在if语句中展开语法正确，保证不管怎么调用都能正确工作。

### 7.6 fcntl 设置非阻塞

```c
// socket 设置非阻塞必须这么做
int flag = fcntl(fd, F_GETFL, 0);
fcntl(fd, F_SETFL, flag | O_NONBLOCK);
```

**为什么需要非阻塞？** → 边缘触发epoll必须配合非阻塞，保证循环读完所有数据不会卡住。

### 7.7 errno 错误码

**系统调用失败后，错误码存在 `errno` 全局变量中：**
- `EAGAIN/EWOULDBLOCK`：非阻塞socket读不到数据了，读完了
- `EINTR`：调用被信号中断，需要重试

项目中处理了这些情况，不会出错。

### 7.8 信号处理 SIGCHLD

**项目中父进程回收子进程：**
```c
signal(SIGCHLD, sigchld_handler);
void sigchld_handler(int signo) {
    while(waitpid(-1, NULL, WNOHANG) > 0);
}
```

`WNOHANG` 不阻塞，回收所有已经结束的子进程，避免僵尸进程。

### 7.9 位运算 哈希表计算索引

```c
// 模运算换成位与，要求容量是2的幂
index = hash & (table->size - 1);
```

比模运算快一点。

### 7.10 Makefile 编译规则

项目分server和client目录，各自有Makefile，`make` 编译生成可执行文件。

## 九、高频面试问题与标准答案

### Q1: epoll 水平触发和边缘触发区别？为什么边缘触发性能更好？

**A：**
- **水平触发LT**：只要socket缓冲区还有数据没读完，epoll_wait每次都会通知你
- **边缘触发ET**：只有缓冲区从没有数据变成有数据（新数据到来）才通知一次
- **性能区别**：边缘触发不会重复通知，减少不必要的系统调用和上下文切换，高并发下性能更好
- **ET使用注意**：必须把socket设为非阻塞，必须循环read直到recv返回-1且errno == EAGAIN，否则会丢数据

### Q2: 什么是惊群效应？你的项目里怎么处理？

**A：**
- 惊群：多个线程/进程都阻塞在accept上，一个新连接进来，内核会唤醒所有阻塞的线程/进程，但只有一个线程能accept成功，其他都会失败，这个唤醒很多无用线程的现象叫惊群
- 现在内核已经优化了，其实只有一个线程会被唤醒，惊群很少发生了
- 我的项目里只有**一个主线程**做epoll和accept，所以根本没有惊群问题

### Q3: 秒传原理是什么？说一下实现流程？

**A：**
- 核心思想：**内容相同的文件只存储一份**，节省服务器存储空间，上传可以瞬间完成
- 流程：
  1. 客户端对要上传的文件计算SHA256哈希
  2. 把文件名、SHA256、文件大小发给服务端
  3. 服务端查全局文件表，如果已经存在这个SHA256 → 直接在用户虚拟目录创建一个引用 → 完成！就是秒传
  4. 如果不存在 → 客户端正常上传文件内容，服务端存储，然后把SHA256记入全局文件表

### Q4: 断点续传原理是什么？

**A：**
- 断点续传就是上传下载断开后，下次可以从上次中断的地方继续传，不用从头开始
- **上传**：客户端记录已经上传了多少字节offset，下次上传把offset发给服务端，服务端从offset位置继续写
- **下载**：客户端记录已经下载了多少字节offset，下次请求把offset发给服务端，服务端从offset位置继续发

### Q5: 数据库连接池为什么要动态扩容？

**A：**
- 初始只创建少量连接（比如3个），大部分时间足够用，节省数据库资源
- 当遇到突发并发流量，所有连接都被占满时，只要还没到最大连接数，就自动新建连接应对突发
- 如果已经到最大连接数，新来的请求排队等待，保护数据库不会被太多连接压垮
- 优点：兼顾了**资源利用率**和**应对突发流量**的能力，比固定大小连接池更灵活

### Q6: 时间轮算法为什么适合这个场景？为什么不用LRU？

**A：**
- 需求就是：空闲连接超过30秒自动断开，所有连接超时时间都是固定的30秒
- 时间轮正好匹配：30个槽，每秒移动一个槽，处理当前槽所有到期连接，实现简单，O(1)批量处理，效率很高
- LRU是淘汰最久未使用，可以处理不同淘汰策略，但这个场景不需要，时间轮更简单高效
- 当然我也知道LRU怎么实现：双向链表+哈希表，get和put都是O(1)，这个场景不需要而已

### Q7: 为什么用mmap？相比read/write好在哪里？

**A：**
- 传统read：`read(fd, buf, size)` → 数据从内核拷贝到用户空间，然后send的时候再从用户空间拷贝到内核socket缓冲区 → 两次拷贝
- mmap：把文件直接映射到用户空间地址，用户可以直接通过指针读写，只需要一次拷贝到socket缓冲区 → 少一次拷贝，性能更好
- mmap使用也方便，断点续传从偏移开始映射很简单

### Q8: 长短任务分离架构好在哪里？

**A：**
- 短任务（pwd/ls/cd这些）很快，几毫秒就能完成，主线程直接执行，不需要进线程池排队，延迟最低
- 长任务（上传下载）是慢IO，比较耗时，如果主线程直接处理会阻塞，没法处理新请求，所以交给线程池异步执行
- 这样设计，短命令延迟很低，长任务也能并发处理，架构清晰，兼顾了延迟和吞吐量

### Q9: 你的项目里哪些地方用了哈希表？解决哈希冲突用什么方法？

**A：**
- 两个地方：1）全局用户表，sockfd找user_info；2）超时队列，sockfd找超时节点
- 解决哈希冲突用**链表法**：每个哈希桶是一个链表，冲突了就挂到链表上
- 查找的时候，找到对应桶，遍历链表找key匹配的节点

### Q10: SHA256 碰撞概率？为什么可以用来判断文件相同？

**A：**
- SHA256输出是256bit，碰撞概率极低，工程上可以认为不同内容就是不同SHA256，所以用来判断文件相同是可靠的

### Q11: 密码为什么要加盐？crypt是什么？

**A：**
- 加盐就是在密码里混入随机盐值，同样的密码加密后结果不同，防止彩虹表撞库破解
- crypt()是Linux提供的密码哈希函数，本身会自动生成盐，生成的哈希包含盐和结果，验证的时候可以正确分离

### Q12: JWT 原理是什么？为什么要用JWT？

**A：**
- JWT是JSON Web Token，把用户信息加密签名生成token，服务端不需要存储，验证签名就行
- 三部分：Header（算法类型）+ Payload（用户信息等数据）+ Signature（签名）
- 优点：无状态，服务端不需要存储会话，扩展方便，多实例部署不需要共享session

### Q13: 项目亮点是什么（自我介绍的时候说）

**A：**
1. **架构清晰简洁**：长短任务分离，主线程处理短任务保证低延迟，线程池处理长任务，删除了原来冗余的双端口设计，代码更简洁，性能更好
2. **功能完整高级**：不仅支持基础文件操作，还支持断点续传（上传下载都支持）、秒传去重、JWT认证，功能接近真实云盘后端
3. **性能优化到位**：epoll边缘触发，动态连接池，哈希表O(1)查找，mmap零拷贝，平均延迟只有0.6ms，能跑1150 QPS
4. **并发安全**：修复了多处竞态条件、double free、哈希覆盖等并发bug，所有共享资源都正确加锁保护，高并发下稳定运行
5. **工程实践完整**：从需求分析、架构设计、编码、测试、性能测试到重构优化，完整的开发流程

### Q14: 什么是粘包？怎么处理的？

**A：**
- TCP是字节流协议，没有消息边界，所以一次recv可能读到多个命令，也可能只读到半个命令，这就是粘包
- 我的项目处理方式：**发送方每个命令结尾加`\r\n`分隔，接收方缓存读到的数据，按`\r\n`切分，一个一个处理**，这样就解决了粘包问题

### Q15: 为什么不用多进程，每个连接一个进程？

**A：**
- 每个进程有独立地址空间，进程切换上下文开销大，连接多了资源占用大，不如线程池轻量
- 线程共享地址空间，切换开销小，能更好利用多核，适合这个场景
