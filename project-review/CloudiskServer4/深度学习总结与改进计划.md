## 一、项目概述

**CloudiskServer** 是一个基于 **C 语言 Linux 原生开发**的高性能多用户网络云盘服务器。

### 核心功能
- ✅ 用户注册/登录（盐值密码加密 + JWT 认证）
- ✅ 虚拟文件系统（`pwd`/`cd`/`ls`/`mkdir`/`rmdir`）
- ✅ 文件上传/下载
- ✅ **断点续传**（上传下载都支持）
- ✅ **秒传功能**（基于 SHA256 去重）
- ✅ **mmap 零拷贝传输**大文件

### 技术栈
| 技术 | 用途 | 优势 |
|------|------|------|
| C11 | 开发语言 | 系统级编程，直接控制内存，无 GC，性能极致 |
| POSIX Socket | 网络通信 | 标准 Linux API，稳定可控 |
| epoll (ET边缘触发) | IO 多路复用 | O(1) 处理活跃连接，支持万级并发 |
| pthread | 线程池 | 成熟稳定的 POSIX 线程库 |
| MySQL C API | 数据库 | 存储元数据，支持事务 |
| l8w8jwt + mbedTLS | JWT 认证 | 无状态令牌认证 |
| crypt() | 密码加密 | 标准 Unix 加密，自带盐值 |
| mmap | 文件传输 | 减少内存拷贝，提升传输性能 |

## 二、架构深入理解

### 2.1 进程架构：父进程 + 子进程（守护进程模式）

```
┌─────────────────────────────────────────────────────────────┐
│                     父进程                                    │
│  - 信号处理(SIGUSR1) 通过管道通知子进程退出                  │
│  - waitpid 回收子进程资源                                    │
└─────────────────────────────┬────────────────────────────────┘
                              │ fork()
                              ▼
┌─────────────────────────────────────────────────────────────┐
│                     子进程（工作进程）                        │
│  - 读取配置文件                                              │
│  - 初始化数据库连接池（支持动态扩容）                        │
│  - 创建并启动长任务线程池                                    │
│  - epoll 监听客户端连接                                      │
│  - 处理所有业务逻辑                                          │
└─────────────────────────────────────────────────────────────┘
```

**设计思路：**
- 父进程只做信号协调，不处理业务
- 子进程崩溃不影响父进程，父进程可以正常回收
- 符合 Unix 守护进程设计风格

### 2.2 请求处理架构：长短任务分离

这是本项目**最重要的架构设计决策**：

```
┌─────────────────────────────────────────────────────────────┐
│                    主线程（epoll 循环）                       │
│              ↓ 读取解析客户端命令                             │
│                                                                  │
│  ┌──────────────┐          ┌────────────────────────┐         │
│  │  短任务       │          │         长任务         │         │
│  │ (pwd/ls/cd)  │          │    (puts/gets 上传下载) │         │
│  │  直接执行     │          │       入队线程池        │         │
│  └──────────────┘          └────────────────────────┘         │
└─────────────────────────────┬──────────────────────────────────┘
                              │
                  ┌───────────┴───────────┐
                  ▼                       ▼
          返回响应                       线程池工作线程
                                      ↓ 从连接池获取数据库连接
                                      ↓ 执行业务逻辑
                                      ↓ 释放连接回池
                                      ↓ 返回响应给客户端
```

**为什么这么设计？**

| 设计决策 | 理由 |
|---------|------|
| **短任务主线程直接执行** | 短任务只需要几 ms，不需要入队排队，延迟最低 |
| **长任务线程池异步执行** | 上传下载是慢 IO，异步执行不阻塞主线程接收新请求 |
| **epoll 边缘触发** | 比水平触发减少不必要唤醒，高并发下性能更好 |
| **非阻塞 Socket** | 配合边缘触发必须，保证循环读完所有数据 |

### 2.3 分层架构

```
客户端
  ↓ TCP 连接
epoll 主线程（长短任务分发）
  ↓
任务队列（生产者消费者）
  ↓
长任务线程池（N 个工作线程）
  ↓
数据库连接池（动态扩容）
  ↓
MySQL 数据库（三张表）
```

## 三、核心模块深度分析

### 3.1 线程池 + 任务队列

**数据结构：**
```c
typedef struct threadpool_s {
    pthread_t * pthreads;     // 线程数组
    int pthreadNum;           // 线程数量
    task_queue_t que;         // 阻塞任务队列
} threadpool_t;

typedef struct task_queue_s {
    task_t * pFront;          // 队头
    task_t * pRear;           // 队尾
    int queSize;              // 队列长度
    pthread_mutex_t mutex;    // 保护队列
    pthread_cond_t cond;      // 条件变量
    int flag;                 // 运行/退出标志
} task_queue_t;
```

**工作流程：**
1. 初始化：预创建 `pthreadNum` 个线程
2. 每个线程循环调用 `taskDeque()` 从队列取任务
3. 队列为空时，线程在条件变量上**阻塞等待**
4. 主线程调用 `taskEnque()` 放入任务，**唤醒**一个等待的线程
5. 线程取出任务执行，执行完释放任务内存

**同步机制：** 生产者消费者模型，互斥锁 + 条件变量

### 3.2 数据库连接池（支持动态扩容）

**数据结构：**
```c
typedef struct {
    MYSQL **connections;    // 连接数组（按最大容量预分配）
    int pool_size;          // 当前已使用连接数
    int max_pool_size;      // 最大连接数限制
    pthread_mutex_t lock;   // 保护池
    pthread_cond_t cond;    // 无连接时等待
    int *available;         // 每个连接是否可用
} ConnectionPool;
```

**动态扩容策略：**
- 初始连接数：由配置文件指定（通常等于线程池大小）
- 最大连接数：配置上限，保护数据库不被压垮
- 当所有连接都被占用，且当前连接数 < 最大连接数 → **自动新建连接**加入池子
- 达到最大连接数后，新来的请求阻塞等待

**为什么需要动态扩容？**
- 日常流量：只需要少量连接，节省资源
- 突发流量：自动扩容应对，提升吞吐量
- 兼顾资源效率和应对突发能力

### 3.3 哈希表优化用户查找

**重构前：** 链表遍历 O(n)，1000 个用户平均需要 500 次比较

**重构后：** 哈希表 O(1) 查找

```c
// sockfd 转字符串作为key
char key[32];
snprintf(key, sizeof(key), "%d", sockfd);
user_info_t *user = (user_info_t*)find(&userTable, key);
```

**哈希表实现细节：**
- 哈希函数：`hashVal = (hashVal << 5) + hashVal + *key++;`（伯恩斯坦哈希）
- 冲突解决：**线性探测法**
- 大小固定：`MAX_SIZE=100`（足够应对连接数限制）

**性能提升：** 1000 并发用户，查找速度提升 **50-100 倍**

### 3.4 超时管理：时间轮算法

**需求：** 30 秒空闲连接自动断开

**数据结构：**
```c
#define TIMEOUT 30          // 30秒超时
#define HASH_SIZE 256       // 哈希映射大小

typedef struct TimeoutQueue_s {
    Node *slots[TIMEOUT];           // 时间轮槽位数组
    MapNode *map[HASH_SIZE];        // sockfd -> 槽位索引的哈希映射
    int currentIndex;               // 当前指针位置
} TimeoutQueue;
```

**工作原理：**
- 时间轮分成 30 个槽，对应 30 秒
- 新连接添加到 `(currentIndex + 29) % 30` 槽位
- epoll 每秒超时一次，指针前进一格，处理当前槽位所有到期连接
- 使用哈希表存储 `sockfd → 槽位索引`，支持 O(1) 查找和更新

**为什么用时间轮而不是排序链表？**
- 到期时间都是整秒，时间轮实现简单
- 移动一次处理一批，效率 O(1) 分摊
- LRU 对于这个场景过于复杂，不需要

### 3.5 解决 TCP 粘包：小火车协议

**设计思想：** 先发送长度，再发送内容

```
+--------+----------+
|  len(4) |  content |
+--------+----------+
```

**为什么能解决粘包？**
- 接收方先读 4 字节得到长度，然后正好读 length 字节
- 每次消息边界清晰，不会粘在一起

**代码保证：** `sendn()` 和 `recvn()` 循环发送/接收直到完成准确长度

### 3.6 秒传功能实现

**流程图：**

```
客户端上传前：
  1. 客户端计算文件 SHA256
  2. 发送 SHA256 给服务器询问是否已存在

服务器检查：
  SELECT id FROM global_file WHERE hash = ?

  if (找到了) {
    // 秒传：直接在虚拟文件表插入记录，不需要传文件
    INSERT INTO virtual_file (...) VALUES (...);
    返回"秒传成功"
  } else {
    // 需要正常上传
    返回"需要上传"，接收文件内容，存储后插入记录
  }
```

**优点：**
- 节省磁盘空间：相同内容多个用户共享只存一份
- 节省上传时间：用户秒传完成
- 节省服务器带宽：不需要重复传输

### 3.7 数据库设计

**三张表设计：**

| 表名 | 用途 |
|------|------|
| `user` | 用户基本信息（id, username, salt, cryptpasswd, path_id）|
| `virtual_file` | 虚拟文件系统（每个用户看到的文件树）|
| `global_file` | 全局文件去重表（存储 SHA256 哈希）|

**虚拟文件系统实现思路：**
- 用数据库模拟目录树，不直接使用磁盘目录
- 每个文件/目录记录 `parent_id` 和完整虚拟路径
- 用户 `path_id` 记录当前工作目录
- 优点：权限控制容易（每个用户只能看到自己的文件），方便实现秒传

## 四、现存问题分析

经过深入代码审查，发现以下潜在问题：

### 4.1 哈希表潜在问题

| 问题 | 描述 | 风险等级 |
|------|------|----------|
| 固定大小限制 | `MAX_SIZE = 100`，超过 100 个并发连接后，`insert` 会拒绝插入 | 🟡 中 |
| 线性探测退化 | 哈希冲突多时，查找退化为 O(n) | 🟡 中 |
| 删除后留下墓碑 | 删除后不会重新探测整理，会导致查找路径中断，查找失败 | 🔴 高 |

**问题详细说明：** 线性探测法删除一个元素后，如果直接标记为空，那么原来探测序列上在它后面的元素就找不到了。因为查找时遇到空就停止了。目前实现就是这个问题。

### 4.2 并发安全问题

| 问题 | 描述 | 风险等级 |
|------|------|----------|
| 时间轮映射没有锁保护 | 主线程 epoll 每秒处理超时，同时用户操作会更新超时，存在竞态 | 🟠 中高 |
| 数据库连接池 `get_connection` 中查找连接是 O(n) | 连接数多了查找变慢，但是这个操作已经拿锁了，并发下锁持有时间变长 | 🟡 中 |
| `userList` 全局链表遍历没有加锁 | 删除节点时，正在遍历会出现野指针 | 🔴 高 |

### 4.3 资源管理问题

| 问题 | 描述 | 风险等级 |
|------|------|----------|
| 连接收缩不支持 | 只支持动态扩容，不支持空闲连接收缩，长期空闲浪费资源 | 🟢 低 |
| 内存分配失败检查不全 | 部分 `malloc` 没有检查返回值，分配失败直接崩溃 | 🟠 中高 |
| SQL 查询结果 `mysql_free_result` 不是所有路径都调用 | 可能造成结果集泄漏，长时间运行积累 | 🟠 中高 |
| `exitPipe` 信号处理使用非异步安全函数 `printf` | 信号处理器中调用 printf 可能造成死锁 | 🟡 中 |

### 4.4 性能问题

| 问题 | 描述 | 风险等级 |
|------|------|----------|
| 路径查询每次都查数据库 | 频繁 cd/ls 每次都要访问 MySQL，有网络开销 | 🟡 中 |
| epoll 数组大小固定 `EPOLL_ARR_SIZE = 100` | 一次最多处理 100 个就绪事件，超过了下次再处理，其实问题不大 | 🟢 低 |
| mmap 每次传输都要重新映射 munmap | 大文件频繁传输有开销 | 🟡 中 |

### 4.5 功能问题

| 问题 | 描述 | 风险等级 |
|------|------|----------|
| 删除文件不删除物理文件 | 当最后一个引用删除后，物理文件还留在磁盘上 | 🟠 中高 |
| 不支持文件重命名 | 基本功能缺失 | 🟠 中 |
| 没有磁盘配额限制 | 用户可以无限上传，占满磁盘 | 🟠 中 |
| 日志功能不完善 | 只有简单的任务日志，错误日志不完整 | 🟡 中 |

## 五、可优化点汇总

### 5.1 架构优化

| 优化点 | 方案 | 预期收益 | 难度 |
|--------|------|----------|------|
| 替换哈希表冲突处理 | 线性探测 → 链式哈希（拉链法） | 消除删除问题，避免冲突退化 | 低 |
| 哈希表可动态扩容 | 固定大小 → 负载因子超过阈值后重新哈希扩容 | 支持更多并发连接，不拒绝插入 | 中 |
| LRU 缓存热点路径 | 缓存最近访问的目录路径信息 | 减少数据库查询，路径查询速度提升 5-10 倍 | 中 |
| 使用 io_uring 替代 epoll | 新一代异步 IO，减少系统调用 | 进一步提升 IO 性能，特别是高并发场景 | 高 |
| 连接池支持收缩 | 空闲连接超过一定时间自动关闭回收 | 长期运行节省资源 | 低 |

### 5.2 性能优化

| 优化点 | 方案 | 预期收益 | 难度 |
|--------|------|----------|------|
| sendfile 零拷贝下载 | mmap → sendfile | 再减少一次内存拷贝，大文件更快 | 低 |
| 大文件分块传输 | 整个文件一次 mmap → 分块 | 超大文件内存占用更平稳 | 中 |
| 预读取缓存 | 小文件预读到内存 | 热点小文件下载更快 | 中 |
| 数据库索引优化 | 给常用查询字段加索引 | 查询速度提升 | 低 |

### 5.3 安全增强

| 优化点 | 方案 | 预期收益 | 难度 |
|--------|------|----------|------|
| TLS/HTTPS 加密传输 | 明文 → OpenSSL 加密 | 防止流量窃听，更安全 | 中高 |
| 密码哈希升级 | crypt → bcrypt/Argon2 | 更安全的密码存储，抵抗暴力破解 | 中 |
| 请求频率限流 | 添加令牌桶限流 | 防止恶意攻击，保护服务器 | 中 |
| 文件类型检查 | 检查文件头，防止上传可执行文件等危险类型 | 安全增强 | 低 |

### 6.5 压力测试

**测试目标：** 找到系统性能瓶颈和极限

**测试方法：**
- 逐渐增加并发连接数，从 10 → 50 → 100 → 500 → 1000
- 每个级别持续压测 5 分钟
- 观察：
  - 错误率是否上升
  - 延迟是否明显增加
  - CPU/内存是否持续增长
  - 是否崩溃

**预期问题发现：**
- 连接数超过哈希表 MAX_SIZE 后的拒绝服务
- 内存泄漏导致 OOM
- 锁竞争导致性能急剧下降

### 6.6 并发测试（竞态条件测试）

**测试目标：** 发现并发安全问题

**测试方法：**
1. **同一个连接并发操作**
   - 多线程同时读写同一个 sockfd
   - 观察是否出现数据损坏

2. **并发添加删除用户**
   - 100 个线程同时添加删除用户
   - 循环 1000 次
   - 检查哈希表状态是否一致，是否出现内存泄漏

3. **连接同时超时删除并发操作**
   - 一个线程操作用户，一个线程时间轮清理
   - 观察是否出现竞态（double free 或野指针）

**竞态分析工具：**
- 使用 ThreadSanitizer 编译运行：`clang -fsanitize=thread ...`
- 自动检测数据竞争

### 6.7 安全测试

**测试清单：**

| 测试项 | 方法 | 预期结果 |
|--------|------|----------|
| SQL 注入 | 用户名输入 `admin' OR '1'='1` | 无法绕过认证 |
| 路径遍历 | 文件名输入 `../../etc/passwd` | 无法访问服务器上其他文件 |
| 缓冲区溢出 | 发送超长用户名/文件名 | 服务器不崩溃，正确拒绝 |
| JWT 伪造 | 修改 JWT payload 篡改用户 id | 服务器验证失败，拒绝连接 |
| 密码暴力测试 | 同一个用户名尝试大量密码 | 服务器没有性能问题（如果有限流更好） |

### 6.8 内存测试

**测试工具：**
- **Valgrind**：检测内存泄漏和非法内存访问
  ```bash
  valgrind --leak-check=full ./CloudiskServer conf/server.conf
  ```
- **AddressSanitizer**：更快发现内存问题
  ```bash
  gcc -fsanitize=address -g ...
  ```
- **malloc 调试**：`mtrace` 跟踪内存分配

**预期：** 长时间运行后，内存增长稳定，没有持续泄漏

这是一个非常优秀的后端面试项目，覆盖了：

- **Linux 网络编程**：socket, epoll, 边缘触发, 非阻塞IO, TCP粘包处理
