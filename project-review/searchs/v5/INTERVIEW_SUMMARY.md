# 搜索引擎项目 面试知识点总结

## 项目整体架构

### 架构分层
1. **离线预处理层**：
   - 文本预处理（英文转小写、中文清洗）
   - 分词统计词频，去停用词
   - 生成前缀词典（用于搜索推荐）
   - Simhash 文章去重
   - 构建倒排索引，计算 TF-IDF 权重
   - 归一化文档权重用于余弦相似度计算

2. **在线服务层**：
   - Reactor 模式处理网络IO
   - 线程池处理计算任务
   - 双缓存 LRU + Redis 二级缓存
   - 搜索推荐（编辑距离排序）
   - 全文搜索（倒排索引 + TF-IDF + 余弦相似度排序）

### 核心流程图
```
用户请求 → 解析HTTP → /suggest 或 /search
             ↓
           一级内存缓存 (Double LRU) 查询
             ↓ 命中 → 直接返回
             ↓ 未命中
           二级 Redis 缓存 查询
             ↓ 命中 → 写入内存缓存 → 返回
             ↓ 未命中
           计算倒排索引/编辑距离 → 写入两级缓存 → 返回
```

## 高频面试问题 & 解答

### 1. 为什么用倒排索引？

**答：**
- 倒排索引是"词 → 文档"的映射，而正排索引是"文档 → 词"
- 用户搜索是**根据词找文档**，倒排索引可以快速找出包含这个词的所有文档
- 同时存储词在文档中的 TF-IDF 权重，方便计算相似度排序

### 2. TF-IDF 原理是什么？公式？

**答：**
- **TF (词频)**：词在当前文档中出现的次数
  - 词频越高，词对文档越重要
- **IDF (逆文档频率)**：log(总文档数 / (包含该词的文档数 + 1)) + 1
  - 一个词出现在越少文档中，IDF越大，说明这个词区分度越高
- **权重 = TF × IDF**

意义：衡量一个词对一篇文档的重要性。常见词（停用词）IDF 小，稀有词 IDF 大。

### 3. 如何计算查询和文档的相似度？

**答：**
使用**余弦相似度**：
- 将查询表示为向量 Q，每个维度是查询词的 TF-IDF 权重
- 将文档表示为向量 D，每个维度是文档中该词的权重
- 余弦相似度 = (Q · D) / (|Q| × |D|)
- 值越大相似度越高，按相似度降序排序返回给用户

### 4. Simhash 去重原理？为什么能处理大规模文档？

**答：**
Simhash 是一种**局部敏感哈希**，可以将一篇文章映射为一个 64bit 指纹：
1. 分词，给每个词计算 hash 得到 64bit
2. 每个位如果是1，加上这个词的权重，如果是0，减去权重
3. 最终每位加权结果 >0 则该位为1，否则为0，得到最终 64bit 指纹
4. 两篇文章海明距离 ≤ 3 认为是重复文章

**优化大规模去重：**
- 性质：如果两个 64bit 指纹海明距离 ≤ 3，那么**最高 8 位一定相同**
- 根据最高 8 位分桶（2^8 = 256 个桶）
- 只需要和同一个桶里的文章比较，不需要和所有文章比较
- 时间复杂度从 O(N²) → O(N × 平均桶大小)，n=10000 从 1 亿次比较降到几千次

### 5. 搜索推荐（suggest）是怎么做的？前缀搜索怎么优化？

**答：**
1. **前缀索引**：把每个词典词条按字拆分，每个字映射到包含该字的所有词条
   - 例如"搜索引擎"拆分"搜"、"索"、"引"、"擎"，每个字的索引都包含这个词的id
   - 用户输入"so"，每个字母找索引求并集，得到所有候选词
2. **排序**：对候选词计算和用户输入的**最小编辑距离**，编辑距离越小越相似，排在前面
3. **编辑距离**：动态规划计算，将一个字符串转换成另一个需要的最少增删改操作次数

### 6. 最小编辑距离动态规划递推公式？

**答：**
`dp[i][j]` 表示 `s1[0..i-1]` 和 `s2[0..j-1]` 的最小编辑距离：
- 边界：`dp[i][0] = i`，`dp[0][j] = j`
- 如果 `s1[i-1] == s2[j-1]`：`dp[i][j] = dp[i-1][j-1]`
- 如果不等：`dp[i][j] = min(dp[i-1][j], dp[i][j-1], dp[i-1][j-1]) + 1`
  - 分别对应删除、插入、替换

**中文处理优化**：按字符（UTF-8 编码一个中文字符 3 字节）计算，正确处理中英文混合

### 7. 缓存架构为什么设计成双缓存 LRU？

**答：**
需求：
- 需要一级内存缓存减少 Redis 访问，提高响应速度
- 高并发场景，尽量减少锁竞争，因为读多写少

双缓存设计：
- 两个缓存：`_current_cache`（前台，服务查询），`_background_cache`（后台，接收新写入）
- 读操作全部访问前台缓存，只有 LRU 移动需要加锁，几乎无锁读
- 新写入都放到后台缓存
- 后台线程定期（比如 60 秒）将后台缓存合并到前台缓存，合并完成清空后台
- 合并只需要一次加锁，大大减少锁竞争时间

相比其他方案：
- **分段锁**：仍然需要每次访问加锁，竞争比双缓存多
- **RCU 读写锁**：读时不加锁，但退占需要同步，实现复杂
- **Thread-local 缓存**：每个线程一份，内存重复占用，同步复杂
- 双缓存实现简单，读几乎无锁，适合这个项目"读多写少，缓存热门查询"的场景

### 8. LRU 淘汰策略原理？

**答：**
- 双向链表 + 哈希表
- 双向链表按访问顺序保存，最近访问放表头，最久未使用放表尾
- 哈希表 O(1) 查找节点
- 访问命中：移动到表头
- 插入：放到表头，如果容量满了删除表尾节点

### 9. Redis 并发问题怎么解决的？

**答：**
hiredis 的 `redisContext` **不是线程安全**的，不能多个线程并发使用同一个连接

解决方案：**thread_local 单例模式**
```cpp
static thread_local RedisSingleton instance;
```
- 每个线程创建自己独立的 Redis 连接
- 每个线程终身使用这个连接
- 完全不需要加锁，天生线程安全
- 空间换时间，并发性能好

### 10. 并发量大的时候，所有数据库连接都被占满怎么办？

**答：**
可以设计**动态扩容连接池**：
- 初始连接数 = 线程数
- 设置最大连接数上限（保护数据库不被冲垮）
- 当请求获取连接时发现无可用连接，且当前连接数 < 最大连接数，**动态新建一个连接**加入池
- 达到最大连接数后才阻塞等待
- 应对突发流量，避免因为连接池满了所有请求卡住

### 11. epoll 水平触发和边缘触发区别？为什么边缘触发性能更好？

**答：**
- **水平触发 LT**：只要 socket 缓冲区有可读数据，epoll 就会一直通知你
- **边缘触发 ET**：只有当缓冲区从"不可读"变成"可读"（即有新数据到来）的时候才通知一次

边缘触发性能更好：
- 不会重复通知已经可读但没读完的 socket，减少系统调用次数
- 使用 ET 需要把 socket 设置为**非阻塞**，循环读直到 `recv` 返回 `-1` 且 `errno == EAGAIN`

### 12. Reactor 模式是什么？这里怎么用的？

**答：**
Reactor 是**事件驱动**的并发网络编程模式：
- 主线程（Reactor）只负责监听 epoll 文件描述符的IO事件
- 有事件到来就把处理任务**分发**给线程池去做
- 优点：分离IO和计算，单线程处理IO避免锁竞争，线程池利用多核并行计算

在这个项目中：
- 主线程 epoll 监听 listenfd 和 clientfd
- 有新连接 → accept → 添加到 epoll
- 客户端有请求可读 → 读取HTTP请求 → 任务扔给线程池处理
- 线程池完成计算 → 回发响应

### 13. 项目中哪里用了分治？哪里用了空间换时间？

**答：**
- **分治**：Simhash 分桶去重，将所有文档分到多个桶，只在桶内比较，原问题分解为子问题
- **空间换时间**：
  - 前缀索引：空间换时间，快速找到所有包含前缀字符的候选词
  - 缓存：内存缓存换 Redis 网络IO时间
  - 倒排索引：预建立词→文档映射，空间换查询时间

### 14. 中文分词用的什么？

**答：**
项目中直接使用了 `cppjieba` 开源库，中文分词准确率高，支持精确模式、全模式等。

### 15. 停用词作用是什么？

**答：**
- 停用词是"的、了、吗、a、an"这类高频但无实际意义的词
- 去停用词可以：
  1. 减少索引大小，节省空间
  2. 减少计算量，提高搜索速度
  3. 避免无意义的词影响 TF-IDF 权重计算，提高搜索准确性

## 项目亮点（面试自我介绍用）

1. **性能优化**：Simhash 去重从 O(N²) 分桶优化到近线性时间，处理万篇文档秒出结果
2. **缓存架构**：设计实现了双缓存 LRU 一级内存缓存，热点查询结果几微秒返回，大大降低 Redis 负载
3. **并发安全**：采用 thread_local 每个线程一个 Redis 连接，彻底解决 hiredis 非线程安全问题，完全无锁并发
4. **架构拆分**：将离线预处理和在线服务分离，原来启动几分钟优化到几秒钟，开发部署更清晰
5. **Bug 修复**：修复了 TF-IDF 计算公式错误、多词查询交集逻辑错误、缓冲区溢出等多个高危 Bug，项目质量大大提高

## 可以延伸的优化方向

1. **BM25 排序**：替换 TF-IDF + 余弦相似度，BM25 在搜索引擎中效果更好
2. **前缀树 Trie**：搜索推荐的前缀匹配可以用 Trie 树更节省空间
3 **反向索引压缩**：倒排索引可以用 Frame of Reference 压缩，节省内存提高cache命中率
4. **结果分页**：当前一次返回所有结果，可以支持分页返回
5. **布隆过滤**：快速判断词是否在索引中
