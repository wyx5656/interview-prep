## 一、项目整体介绍

### 1.1 项目做了什么

这是一个**完整的C++搜索引擎项目**，包含：

- **离线阶段**：对几千篇RSS网页进行预处理，去重、分词、构建倒排索引
- **在线阶段**：启动HTTP服务器，提供两个接口：
  1. `/suggest` - 搜索词推荐（用户输入前缀，返回相似候选词按编辑距离排序）
  2. `/search` - 全文搜索（返回相关文档按相似度排序）

### 1.2 技术栈

| 模块 | 技术 | 说明 |
|------|------|------|
| 分词 | `cppjieba` | 开源中文分词库 |
| 去重 | Simhash | 局部敏感哈希，海明距离判断重复 |
| 索引 | 倒排索引 | 词 → 文档列表 + TF-IDF权重 |
| 排序 | TF-IDF + 余弦相似度 | 计算查询和文档的相似度 |
| 搜索推荐 | 前缀索引 + 最小编辑距离 | 候选词匹配 + 排序 |
| 缓存 | 双缓存LRU（一级内存） + Redis（二级） | 热点查询加速 |
| 网络 | Reactor + epoll边缘触发 + 线程池 | 并发网络IO |
| Redis并发 | thread_local 每个线程一个连接 | 解决hiredis非线程安全问题 |

```
│  - 结果写入缓存 → 回写HTTP响应                                    │
└─────────────────────────────────────────────────────────────────────┘

```

## 三、离线预处理模块详解

### 3.1 TextProcessor - 词典预处理

**文件位置：** `include/ProcessFile.h` `src/ProcessFile.cc`

**功能：**
1. 读取英文词典和中文词典
2. 统计词频
3. 构建前缀索引用于搜索推荐
4. 去停用词
5. 提供最小编辑距离计算方法

**核心成员：**
```cpp
std::unordered_map<std::string, int> _dict;         // 词 -> id
std::unordered_map<int, std::string> _idft;         // id -> 词
std::unordered_map<char, std::unordered_set<int>> _index; // 字符 -> 包含该字符的词id集合
```

**前缀索引原理：**
- 对每个词，拆分成一个个字符
- 每个字符，在索引中添加这个词的id
- 用户输入前缀如 "sear"：
  - 对每个字符's','e','a','r'分别取出词id集合
  - 求交集，得到同时包含这四个字符的所有词 → 就是候选词
- 时间复杂度：预处理建索引O(总字符数)，查询O(集合交集)

**最小编辑距离：**
- 动态规划，定义 `dp[i][j]` = s1[0..i-1] 和 s2[0..j-1] 的最小编辑距离
- 详细公式见第5章

### 3.2 RSS - 网页解析

**文件位置：** `include/RSS.h` `src/RSS.cc`

**功能：**
1. 遍历目录下所有RSS XML文件
2. 使用`tinyxml2`库解析XML
3. 提取每个item的：title、link、content
4. 存储清洗后的网页数据

**存储结构：** `vector<RSSItem> _items;`

### 3.3 webPageHandler - 倒排索引构建

**文件位置：** `include/webPageHandler.h` `src/webPageHandler.cc`

**核心静态成员：**
```cpp
static std::vector<std::pair<int, int>> offsetlib; // docid -> (offset, length)
static std::unordered_map<std::string, std::set<std::pair<int, double>>> newnewindex;
// 倒排索引：词 -> set(docid, weight)
```

**主要步骤：**

1. **getoffsetlib()** - 生成偏移表，Simhash去重
   - 读取所有网页
   - 每个网页计算Simhash指纹
   - 按最高8位分桶（256个桶）
   - 只和同桶比较海明距离，去掉重复文档
   - 存储去重后的文档偏移信息到偏移文件

2. **generateIndex()** - 生成倒排索引
   - 遍历每个不重复文档
   - jieba分词，去停用词
   - 统计每个词在文档中的词频TF
   - 计算IDF = log(N / DF)，N是总文档数，DF是包含该词的文档数
   - 权重 = TF × IDF
   - 构建倒排索引 `词 -> set(docid, weight)`
   - 归一化（详见下文）
   - 保存到文件

3. **getwebPage()** - 根据docid读取网页内容
   - 从偏移表得到offset和length
   - 从网页库文件读取指定长度
   - 返回webPage对象

**Bug修复记录：**
- 原代码用变长数组 `char buffer[len]` 在栈上分配 → 大文档栈溢出 → 修复为 `vector<char>` 堆分配

### 3.4 Dictonary - 词典单例

## 四、在线服务网络模块详解

### 4.1 Reactor 模式设计

**核心思想：** 分离IO和计算
- 主线程只负责监听epoll事件
- 有IO事件就把任务分发到线程池
- 工作线程负责计算，计算完回写响应

**优点：**
- 主线程不阻塞，避免不必要的锁竞争
- 线程池利用多核并行计算
- 比多进程/多线程每个连接一个线程节省资源

### 4.2 epoll 边缘触发 ET vs 水平触发 LT

| 对比 | 水平触发 LT | 边缘触发 ET |
|------|------------|-------------|
| 通知时机 | 只要缓冲区有数据就一直通知 | 只有从无变有（新数据到来）才通知一次 |
| 系统调用 | 多次 | 一次 |
| 性能 | 较低 | 较高 |
| 要求 | 可以阻塞读 | 必须非阻塞读，循环读到EAGAIN |

**项目中使用：** `epoll_event.events = EPOLLIN | EPOLLET | EPOLLONESHOT;`

### 4.3 主要类关系

**Echoserver** - 服务器主入口
```cpp
Echoserver server(4, 10, "0.0.0.0", 8888);
// 参数：线程数、超时、IP、端口
server.start();
```

**TcpServer** - 持有Acceptor和EventLoop

**Acceptor** - 监听socket，accept新连接

**EventLoop** - Reactor主循环
```cpp
void loop() {
    while (!_quit) {
        int number = epoll_wait(_epollfd, events, 1000, timeout);
        for (int i = 0; i < number; ++i) {
            handleEvent(events[i]);
        }
    }
}
```

**TcpConnection** - 一个TCP连接的封装
- `handleRead()` - 读取HTTP请求
- `handleWrite()` - 发送HTTP响应

**ThreadPool** - 线程池
- 生产者消费者模型，任务队列 + 互斥锁 + 条件变量
- `addTask()` - 添加任务
- 工作线程循环取任务执行

**HTTPRequest** - HTTP请求解析
- 解析请求行、请求头
- 得到method、path、body

**HTTPResponse** - HTTP响应组装
- 组装状态行、头、body

**HTTPRequestHandler** - 处理具体业务
- `handleSuggest()` - 处理搜索推荐
- `handleSearch()` - 处理全文搜索
- `getResultFromCacheOrBackend()` - 查询缓存

### 4.4 一次完整请求处理流程

1. 客户端connect → 内核通知epoll → accept得到clientfd → 创建TcpConnection → 添加到epoll
2. 客户端send请求 → 内核通知epoll可读 → TcpConnection::handleRead → 读取数据 → 解析HTTP → 封装task → 线程池addTask
3. 工作线程取出task → 调用HTTPRequestHandler::onPost → 根据path调用handleSuggest/handleSearch
4. 查询一级DoubleLRU缓存 → 命中直接返回JSON → 不命中查询Redis → 命中返回写入缓存 → 不命中计算结果写入缓存
5. 得到结果后 → TcpConnection::send() → epoll检测到可写 → handleWrite发送响应 → 发送完成关闭连接

## 五、核心算法完全解析

### 5.1 Simhash 文档去重

**问题：** 几千篇文档，找出内容重复相似的文档，去掉重复。

**原理：** 局部敏感哈希 → 相似的文档得到相似的64bit指纹。

**算法步骤：**

```
1. 分词 → 每个词得到权重w
2. 对每个词，计算64bit哈希值hash
3. 对64bit的每一位：
   如果该位是1 → 累加w到该位
   如果该位是0 → 减去w从该位
4. 全部词处理完后，对每一位：
   如果结果 > 0 → 该位为1
   如果结果 ≤ 0 → 该位为0
5. 得到最终64bit指纹
```

**判断重复：** 计算两个指纹的**海明距离**（不同bit的个数），如果距离 ≤ 3 → 认为重复。

**大规模优化：**

```
性质：如果海明距离 ≤ 3 → 最高 8 位一定相同
原因：总共只改变≤3bit，最高8bit不可能全变
所以：
- 按最高8位分桶，总共 2^8 = 256 个桶
- 一个新文档只需要和同桶的文档比较
- 不需要和所有文档比较
时间复杂度：从 O(N²) → O(N × 平均桶大小)
N=10000，从 1亿次比较 → 几千次比较
```

**代码位置：** `webPageHandler::getoffsetlib()`

### 5.2 TF-IDF 权重计算

**定义：**
- **TF（词频）** = 词在当前文档中出现的次数
- **DF（文档频率）** = 有多少个文档包含这个词
- **IDF（逆文档频率）** = `log( N / (DF + 1) ) + 1`，N = 总文档数

**权重公式：** `weight = TF × IDF`

**意义：**
- 常用停用词 → DF大 → IDF小 → 权重小
- 稀有词 → DF小 → IDF大 → 权重大
- 这样稀有词对搜索更重要，符合直觉

**代码位置：** `webPageHandler::generateIndex()` 行 262-276

### 5.3 倒排索引

**结构：** `unordered_map<string, set<pair<int, double>>>` → 词 → 集合(文档id, 权重)

**为什么叫倒排：**
- 正排索引：文档id → 文档包含的词
- 倒排索引：词 → 包含这个词的文档
- 用户搜索是输入词找文档，所以倒排索引更合适

**代码位置：** `webPageHandler::newnewindex`

### 5.4 余弦相似度排序

**原理：** 把查询和每个文档都表示为向量，计算两个向量的夹角余弦，余弦越大越相似。

**公式：**

```
相似度 = (Q · D) / (|Q| × |D|)

Q: 查询向量，Q[i] = 查询词i的TF-IDF
D: 文档向量，D[i] = 文档中词i的TF-IDF

Q · D = 累加 Q[i] × D[i]
|Q| = sqrt(累加 Q[i]²)
|D| = sqrt(累加 D[i]²)
```

**优化：** 预处理的时候已经对每个文档的权重做了归一化，使得 `|D| = 1`，所以只需要算 `(Q · D) / |Q|`，节省计算时间。

**归一化代码位置：** `webPageHandler::normalize()`

**排序代码位置：** `HTTPRequestHandler::handleSearch()`

### 5.5 最小编辑距离（搜索推荐）

**定义：** 将字符串s1转换为s2最少需要多少次操作（增、删、改）。越小越相似。

**动态规划：**

`dp[i][j]` = s1[0..i-1] 和 s2[0..j-1] 的最小编辑距离

**边界条件：**
```
dp[i][0] = i  // s2是空，删除i个字符
dp[0][j] = j  // s1是空，插入j个字符
```

**递推公式：**
```
如果 s1[i-1] == s2[j-1]:
    dp[i][j] = dp[i-1][j-1]  // 最后一个字符相同，不需要操作
否则:
    dp[i][j] = min(
        dp[i-1][j],     // 删除：删除s1[i-1]，问题变成s1[0..i-2]和s2[0..j-1]
        dp[i][j-1],     // 插入：插入s2[j-1]，问题变成s1[0..i-1]和s2[0..j-2]
        dp[i-1][j-1]    // 替换：把s1[i-1]替换成s2[j-1]，问题变成s1[0..i-2]和s2[0..j-2]
    ) + 1
```

**时间复杂度：** O(len(s1) × len(s2))

**应用：** 用户输入前缀，所有候选词计算编辑距离，从小到大排序，返回前N个。

**代码位置：** `TextProcessor::editDistance()`

### 5.6 LRU 缓存淘汰

**问题：** 缓存容量满了，淘汰哪个？LRU淘汰**最久未使用**的。

**数据结构：** `双向链表 + 哈希表`

```
双向链表：
  表头 ← 最近访问
  表尾 → 最久未使用

哈希表：
  key → 缓存节点，O(1)查找
```

**操作：**
- **访问命中：** 将节点移动到表头
- **插入：** 插入到表头，如果容量满了删除表尾节点

**代码位置：** `ThreadLocalCache` in `DoubleLruCache.h`

## 六、缓存架构设计详解

### 6.1 二级缓存架构

```
用户查询
    ↓
一级缓存：DoubleLRU 每个线程私有内存缓存
    ↓ 命中？ → 是 → 直接返回
    ↓ 否
二级缓存：Redis 网络缓存
    ↓ 命中？ → 是 → 写入一级缓存 → 返回
    ↓ 否
计算得到结果 → 写入一级和二级 → 返回
```

**设计理由：**
- 内存缓存比Redis快一个数量级
- Redis防止整个缓存雪崩，如果一级缓存miss，Redis大概率命中
- 两级缓存，兼顾速度和容量

### 6.2 双缓存 LRU 设计思想

**需求背景：**
- 一级内存缓存，减少Redis访问
- 读多写少，要尽量**减少锁竞争**，提高读性能

**原方案问题：**
- 全局锁：所有读写都要锁，读多情况下锁竞争严重，性能差
- 分段锁：还是每次访问都要加锁，有竞争

**双缓存方案：**

```
每个线程：
  working 缓存 ← 所有读都在这里，********完全无锁********
  sync    缓存 ← 接收新数据同步

全局：
  shared_buffer ← 所有新写入先放到这里，需要锁

同步流程（每个线程读的时候自己检查是否需要同步）：
1. 如果 shared_buffer 为空 → 不同步
2. 如果 shared_buffer 不为空 OR 到同步间隔时间 → 需要同步：
   a. 把当前 working 完整拷贝到 sync
   b. 把 shared_buffer 所有新数据插入到 sync（LRU自动处理淘汰）
   c. 原子交换 working 和 sync
   d. 现在 working 就是完整的新版本（旧数据 + 新数据）
   e. 清空 shared_buffer

为什么读完全无锁：
- 所有读操作都在 working 上，working 只被当前线程访问
- 完全不需要加锁！！！
- 只有同步的时候需要一次加锁，同步是批量的，锁次数大大减少
```

**对比其他方案：**

| 方案 | 读加锁 | 缺点 | 双缓存优点 |
|------|---------|------|-----------|
| 全局锁 | 每次都要 | 竞争严重 | 读完全无锁 |
| 分段锁 | 每次都要 | 还是有竞争 | 只有批量同步加一次锁 |
| RCU读写锁 | 读不加锁 | 实现复杂，退占同步麻烦 | 实现简单 |
| thread-local | 每个线程一份 | 内存重复占用 | 每个线程一份，但容量可控 |

**代码位置：** `DoubleLruCache.h/cc`

**本次修复的bug：**

| Bug | 修复 |
|-----|------|
| 不到同步间隔即使有新数据也不同步 → 新数据长时间读不到 | 有数据就必须同步 |
| 交换后清空sync → 旧数据全部丢失 | 不需要清空，下次拷贝working到sync |
| 拷贝working顺序颠倒 → LRU顺序错了 → 淘汰错对象 | 逆序遍历保持顺序 |
| 缺少reset → 单元测试状态不干净 | 添加reset方法 |

### 6.3 Redis 并发安全解决方案

**问题：** hiredis 的 `redisContext` **不是线程安全**，多个线程不能共用同一个连接。

**解决方案：** `thread_local` 每个线程一个连接

```cpp
static thread_local RedisSingleton instance;
```

- 每个线程创建自己独立的Redis连接
- 每个线程终身使用这个连接
- **完全不需要加锁**，天生线程安全
- 空间换时间，并发性能好

**代码位置：** `RedisSingleton.h/cc`

## 七、项目中用到的C++语法详解

这里列出项目中用到的所有C++语法知识点，每个都有项目中的实际例子：

### 7.1 单例模式

**写法（项目中用的）：**
```cpp
static Dictonary* getInstance() {
    static Dictonary instance;  // 静态局部变量
    return &instance;
}
```

**要点：**
- C++11之后，静态局部变量初始化是**线程安全**的
- 不需要自己写双重检查锁，编译器保证
- 优点：写法简单，懒汉式，不用不用不初始化

**项目中哪里用：** `Dictonary` `DoubleLruCache` 都是单例

### 7.2 禁止拷贝 继承 NonCopyable

**写法：**
```cpp
class NonCopyable {
public:
    NonCopyable() = default;
    NonCopyable(const NonCopyable&) = delete;
    NonCopyable& operator=(const NonCopyable&) = delete;
};
```

**`=delete` 关键字：** C++11新特性，显示删除拷贝构造和赋值，禁止拷贝。

**为什么要用：** Acceptor、EventLoop这些对象不该被拷贝，继承NonCopyable就自动禁止了，不用自己写。

**项目中哪里用：** 很多网络类都继承了。

**写法：**
```cpp
DoubleLruCache() = default;
```

**作用：** 你写了其他构造函数，编译器就不生成默认构造了，用 `=default` 要求编译器仍然生成默认构造。

### 7.4 `nullptr` 代替 NULL

**优点：** `nullptr` 是真正的空指针，类型是 `nullptr_t`，不会和整数混淆，类型安全。项目中都用 `nullptr` 不用 `NULL`。

### 7.5 智能指针 `unique_ptr` `shared_ptr`

**`unique_ptr` 独占所有权：**
```cpp
static thread_local std::unique_ptr<ThreadDoubleCache> thread_cache_;
```
- 独占所有权，不需要释放，离开作用域自动释放
- 项目中用在thread_local每个线程的缓存对象

**`shared_ptr` 共享所有权：**

```cpp
std::shared_ptr<Dictonary> d = Dictonary::getInstance();
```

- 引用计数，最后一个持有者释放自动删除

### 7.6 `thread_local` 线程局部存储

**语法：**

## 八、并发编程要点

### 8.1 生产者消费者模型

**实现：**
```cpp
// 生产者（主线程）
void addTask(function<void()> f) {
    {
        unique_lock<mutex> lock(_mutex);
        _taskQueue.push(f);
    }
    _cond.notify_one();  // 通知一个等待的消费者
}

// 消费者（工作线程）
function<void()> take() {
    unique_lock<mutex> lock(_mutex);
    _cond.wait(lock, [this] { return !_taskQueue.empty(); });
    auto task = _taskQueue.front();
    _taskQueue.pop();
    return task;
}
```

**要点：**
- `wait` 为什么要放在while循环？**虚假唤醒** → 多个消费者同时唤醒，一个拿走了任务，另一个醒来队列又空了，所以要继续等。正确写法：
```cpp
while (_taskQueue.empty()) {
    _cond.wait(lock);
}
```
- lambda谓词给wait，内部已经循环了，所以不用写外面while也可以

项目中 `TaskQueue` 就是这么实现的。

### 8.2 线程安全总结（项目中各个模块怎么保证线程安全）

| 模块 | 线程安全措施 |
|------|-------------|
| 倒排索引词典 | 启动加载完就只读了，天然线程安全 |
| DoubleLruCache读 | working每个线程私有，完全无锁 |
| DoubleLruCache写 | 全局锁保护shared_buffer |
| Redis | thread_local每个线程一个连接，无锁 |
| 任务队列 | mutex + condition_variable 保护 |
| epoll | 只有主线程操作，没有并发问题 |

**项目特点：读多写少，大部分数据只读，所以并发性能很好**

### 8.3 内存可见性

- 共享变量用 `std::atomic` → 保证内存可见性
- 锁和原子操作都提供内存栅栏，保证可见性

## 十、高频面试问题与标准答案

### Q1: 为什么用倒排索引？

**A：**
- 倒排索引是"词 → 文档"的映射，而正排索引是"文档 → 词"
- 用户搜索就是**根据关键词找文档**，倒排索引正好匹配这个需求，可以快速找出包含关键词的所有文档
- 同时存储了每个词在文档中的TF-IDF权重，方便计算相似度排序

### Q2: TF-IDF 原理是什么？公式是什么？

**A：**
- **TF（词频）**：词在当前文档中出现的次数，次数越多越重要
- **DF（文档频率）**：有多少个文档包含这个词
- **IDF（逆文档频率）**：公式 `IDF = log(N / (DF + 1)) + 1`，N是总文档数
  - 一个词出现在越少文档中 → DF越小 → IDF越大 → 说明这个词区分度越高
  - 常见停用词 DF大 → IDF小 → 权重小，符合预期
- **最终权重 = TF × IDF**

### Q3: 余弦相似度公式是什么？为什么要用余弦相似度？

**A：**
- 公式：`相似度 = (Q · D) / (|Q| × |D|)`
  - Q是查询向量，每个维度是查询词的TF-IDF
  - D是文档向量，每个维度是文档中该词的TF-IDF
- 余弦相似度衡量两个向量方向的夹角，余弦越大夹角越小，越相似
- 项目中提前对文档权重做了归一化，`|D| = 1`，所以只需要计算 `(Q · D) / |Q|`，节省计算时间

### Q4: Simhash 去重原理？为什么能处理大规模文档？

**A：**
- Simhash是局部敏感哈希，**相似文档得到相似的64bit指纹**
- 步骤：分词得到每个词和权重 → 每个词hash得到64bit → 每一位加权累加，大于0为1否则为0 → 得到最终64bit指纹
- 判断重复：海明距离 ≤ 3认为重复
- 优化大规模：因为海明距离≤3 → 最高8位一定相同 → 按最高8位分256个桶 → 只需要和同桶文章比较 → 时间复杂度从O(N²)降到近似线性

### Q5: 搜索推荐（suggest）是怎么做的？前缀搜索怎么优化？

**A：**
1. **预处理前缀索引**：每个词拆分成字符，每个字符对应所有包含该字符的词id集合
2. **查询时**：用户输入前缀每个字符，取出对应集合求交集，得到候选词
3. **排序**：对候选词计算和用户输入的最小编辑距离，编辑距离越小越相似，排序返回前N个
4. 空间换时间，预处理建索引，查询很快

### Q6: 最小编辑距离动态规划递推公式？

**A：**
`dp[i][j]`表示 `s1[0..i-1]` 和 `s2[0..j-1]` 的最小编辑距离
```
边界：dp[i][0] = i，dp[0][j] = j
如果 s1[i-1] == s2[j-1]:
  dp[i][j] = dp[i-1][j-1]
否则:
  dp[i][j] = min(dp[i-1][j], dp[i][j-1], dp[i-1][j-1]) + 1
// dp[i-1][j] → 删除s1[i-1]
// dp[i][j-1] → 插入s2[j-1]
// dp[i-1][j-1] → 替换s1[i-1]为s2[j-1]
```

### Q7: 为什么设计双缓存LRU？原理说一下？

**A：**
- 需求：一级内存缓存减少Redis访问，读多写少，要尽量减少锁竞争
- 设计：每个线程两个缓存 working（服务查询）+ sync（同步接收新数据），全局shared_buffer收集新写入
- 同步：每个线程读的时候自己检查，有新数据就同步 → 拷贝working到sync，插入新数据，交换working和sync
- **读完全在working上，完全不需要加锁！** 只有同步需要一次加锁，锁竞争大大减少
- 对比分段锁：分段锁每次读都要加锁，还是有竞争，双缓存读完全无锁，性能更好

### Q8: LRU 原理？数据结构？

**A：**
- 双向链表 + 哈希表
- 双向链表按访问顺序保存：表头最近访问，表尾最久未使用
- 哈希表 O(1) 查找节点
- 访问命中：移动到表头
- 插入：放到表头，容量满了删除表尾

### Q9: Redis 并发问题怎么解决？

**A：**
- hiredis 的 redisContext 不是线程安全，多个线程不能共用同一个连接
- 解决方案：**thread_local 单例模式**，每个线程创建自己独立的Redis连接
- 每个线程终身使用这个连接，**完全不需要加锁**，天生线程安全
- 空间换时间，并发性能好

### Q10: epoll 水平触发和边缘触发区别？为什么边缘触发性能更好？

**A：**
- **水平触发LT**：只要socket缓冲区有数据可读，epoll_wait每次都会通知
- **边缘触发ET**：只有缓冲区从不可读变成可读（新数据到来）才通知一次
- 边缘触发性能更好：不会重复通知已经可读但没读完的socket，减少系统调用次数
- 使用ET需要把socket设为**非阻塞**，循环read直到recv返回-1且errno == EAGAIN

### Q11: Reactor 模式是什么？项目里怎么用的？

**A：**
- Reactor是**事件驱动**的并发网络编程模式
- 主线程（Reactor）只负责epoll监听IO事件，有事件到来就把任务分发到线程池
- 优点：分离IO和计算，主线程只处理IO，不阻塞，线程池利用多核并行计算
- 项目中：主线程epoll监听，有请求就读取解析扔给线程池，线程池计算完回发响应

### Q12: 项目中哪里用了分治？哪里用了空间换时间？

**A：**
- **分治**：Simhash分桶去重，把所有文档分到多个桶，只在桶内比较，原问题分解为子问题，大大减少比较次数
- **空间换时间**：
  1. 前缀索引：预处理建索引，空间换查询时间，前缀查询很快
  2. 缓存：内存缓存换Redis网络IO时间，查询更快
  3. 倒排索引：预建立词→文档映射，空间换查询时间

### Q13: 停用词作用是什么？

**A：**
- 停用词是"的、了、吗、a、an"这种**高频但没有实际语义**的词
- 作用：
  1. 减少索引大小，节省内存空间
  2. 减少计算量，提高搜索速度
  3. 避免无意义的词影响TF-IDF权重计算，提高搜索准确性

### Q14: 项目亮点是什么（自我介绍）

**A：**
1. **算法优化**：Simhash去重通过分桶从O(N²)优化到近似线性，万篇文档秒出结果
2. **缓存架构**：设计实现了双缓存LRU一级内存缓存，热点查询读完全无锁，微秒级返回，大大降低Redis负载
3. **并发安全**：采用thread_local每个线程一个Redis连接，彻底解决hiredis非线程安全问题，完全无锁并发，性能好
4. **架构清晰**：离线预处理和在线服务分离，启动只需要加载预处理好的数据，几秒钟就能启动，开发部署清晰
5. **健壮性**：修复了栈溢出、同步逻辑错误等多个严重bug，代码更安全可靠
