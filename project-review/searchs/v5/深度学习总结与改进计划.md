## 一、项目概述

**搜索引擎 v5** 是一个 **C++ 实现**的完整**全文搜索引擎服务器**，支持中英文搜索和搜索词推荐。

### 核心功能
- ✅ **搜索词推荐（Suggest）**：基于前缀匹配 + 编辑距离排序
- ✅ **全文搜索（Search）**：倒排索引 + TF-IDF + 余弦相似度排序
- ✅ **Simhash 文档去重**：局部敏感哈希 + 分桶优化，海明距离判断
- ✅ **二级缓存架构**：双缓存 LRU 内存缓存 + Redis 持久化缓存
- ✅ **Reactor 网络架构**：基于 epoll 的反应堆模式 + 线程池
- ✅ **支持中英文**：cppjieba 中文分词，支持中英混合搜索
- ✅ **JSON API**：HTTP 接口，方便前端调用

### 技术栈
| 技术 | 用途 | 选型理由 |
|------|------|----------|
| C++11 | 开发语言 | 高性能，直接控制内存，适合搜索引擎计算密集场景 |
| Reactor + epoll | 网络IO | 事件驱动，分离IO和计算，高并发性能好 |
| 线程池 | 任务执行 | 利用多核并行计算，主线程只负责IO |
| cppjieba | 中文分词 | 开源成熟，分词准确率高 |
| Simhash | 文档去重 | 局部敏感哈希，适合大规模文档去重 |
| Redis | 二级缓存 | 持久化缓存，减轻计算压力 |
| nlohmann/json | JSON处理 | 头文件Only，轻量易用 |
| tinyxml2 | XML解析 | 解析RSS订阅源，轻量 |

## 二、整体架构深入理解

### 2.1 项目架构：离线预处理 + 在线服务分离

这是重构后的优秀架构设计：

```
┌─────────────────────────────────────────────────────────────┐
│                   离线预处理 (bin/offline)                     │
│  只需要运行一次，结果保存磁盘，启动在线服务直接加载             │
│                                                               │
│  1. 文本预处理：英文转小写，中文清洗，统计词频                │
│  2. 生成前缀词典：用于搜索推荐前缀匹配                        │
│  3. RSS网页清洗：提取标题、链接、内容                          │
│  4. Simhash去重：移除重复文章                                  │
│  5. 构建倒排索引：计算TF-IDF权重，归一化                       │
│  6. 保存结果到磁盘：偏移表 + 倒排索引                          │
└─────────────────────────────┬─────────────────────────────────┘
                              │ 预处理完成
                              ▼
┌─────────────────────────────────────────────────────────────┐
│                   在线服务 (bin/online)                        │
│  启动快速（几秒钟），直接从磁盘加载索引，处理用户查询           │
└─────────────────────────────┬─────────────────────────────────┘
                              │ 用户请求
                              ▼
┌─────────────────────────────────────────────────────────────┐
│                Reactor 主线程 (epoll 监听)                    │
│  - 接受新连接                                                 │
│  - 读取HTTP请求                                              │
│  - 解析请求，分发任务到线程池                                 │
└─────────────────────────────┬─────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│                         线程池                                │
│  ┌─────────────────────────────────────────────────────────┐│
│  │ 工作线程N个：每个线程处理一个搜索请求                      ││
│  │                                                           ││
│  │  查询流程：                                               ││
│  │   1. 查询一级 双缓存LRU 内存缓存  → 命中直接返回          ││
│  │   2. 未命中 → 查询二级 Redis 缓存 → 命中写入内存返回       ││
│  │   3. 都未命中 → 计算倒排索引/编辑距离 → 写入两级缓存返回   ││
│  └─────────────────────────────────────────────────────────┘│
└─────────────────────────────┬─────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│                   JSON 响应返回客户端                         │
└─────────────────────────────────────────────────────────────┘
```

**为什么分离离线和在线？**
- 预处理只需要做一次，非常耗时（分词、去重、建索引）
- 在线服务启动只需要加载索引，几秒钟就能启动
- 开发部署清晰，修改在线代码不需要重新预处理
- 重启服务快，不影响可用性

### 2.2 请求处理流程

```
用户发送 POST /suggest 或 /search
             ↓
主线程 epoll 触发可读事件
             ↓
读取并解析 HTTP 请求 → 解析 JSON 请求体
             ↓
任务扔给线程池处理
             ↓
工作线程取出任务执行：

  ┌─────────────────────────────────────────────────────────┐
  │  一级缓存查询：DoubleLruCache::get()                      │
  │  → 命中：直接返回 JSON 响应                               │
  │  → 未命中：继续                                           │
  └─────────────────────────────────────────────────────────┘
                               ↓
  ┌─────────────────────────────────────────────────────────┐
  │  二级缓存查询：Redis GET                                  │
  │  → 命中：写入一级缓存 → 返回响应                          │
  │  → 未命中：继续计算                                       │
  └─────────────────────────────────────────────────────────┘
                               ↓
  ┌─────────────────────────────────────────────────────────┐
  │  计算：                                                  │
  │  /suggest: 前缀索引找候选 → 编辑距离排序 → 取topN        │
  │  /search: 倒排索引交集 → TF-IDF + 余弦相似度排序 → topN  │
  └─────────────────────────────────────────────────────────┘
                               ↓
  写入结果到一级缓存和 Redis
                               ↓
  返回 JSON 响应
```

### 2.3 缓存架构：双缓存 LRU + Redis 二级缓存

**设计思路：**
- **一级缓存**：每个线程私有双缓存，存储热门查询结果
- **二级缓存**：Redis 持久化缓存，存储所有查询结果
- **目标**：热门查询结果几微秒返回，减少 Redis 网络IO，降低负载

**架构细节：**
```cpp
// 每个线程私有：
struct ThreadDoubleCache {
    ThreadLocalCache working;    // 前台缓存：处理读查询
    ThreadLocalCache sync;       // 后台缓存：接收新写入
};

// 全局：
std::map<std::string, json> shared_buffer_;  // 收集所有新写入
std::thread sync_thread_;                     // 后台同步线程

// 同步流程：
// 1. 读查询：都在当前线程 working 缓存，无锁（因为线程私有）
// 2. 新写入：写入全局 shared_buffer（需要加锁）
// 3. 后台线程定期（比如 60 秒）唤醒，把 shared_buffer 分发到
//    每个线程的 sync 缓存，然后合并到 working 缓存
```

**为什么这么设计？**
- 读操作：线程私有数据，完全不需要锁，性能极高
- 写操作：只有写入全局缓冲区需要加锁，锁持有时间极短
- 批量同步：定期一次批量合并，减少锁竞争
- 非常适合**读多写少**的缓存场景（搜索查询正好符合）

## 三、核心模块深度分析

### 3.1 倒排索引 + TF-IDF + 余弦相似度

**倒排索引结构：**
```cpp
// 词 -> 文档集合 + 权重
unordered_map<string, set<pair<int, double>>> newnewindex;
// key: 词条
// value: 集合，每个元素是 (docid, 权重)
```

**TF-IDF 计算公式：**
- **TF (词频)**：词在文档中出现的次数
- **DF (文档频率)**：有多少个文档包含这个词
- **IDF = log2(N / (DF + 1)) + 1**，N 是总文档数
- **权重 = TF × IDF**

**归一化：** 每个文档的所有词权重平方和开根号归一化，方便余弦相似度计算

**查询处理（多关键词）：**
1. 对查询分词，去停用词，计算每个查询词的 TF-IDF 权重
2. 对每个查询词，从倒排索引得到包含该词的所有文档
3. 计算**交集**：只保留**包含所有查询词**的文档
4. 对每个候选文档，计算**余弦相似度**：
   ```
   cos_sim = (query_vector · doc_vector) / (|query| × |doc|)
   ```
5. 按余弦相似度降序排序，返回 topN 给用户

### 3.2 搜索推荐（Suggest）前缀匹配 + 编辑距离

**前缀索引结构：**
```cpp
// 词典中每个词 "搜索引擎"
// 拆分每个字符："搜"、"索"、"引"、"擎"
// 每个字符对应的 map 都加入这个词的索引
map<string, set<int>> _Chineseindex;
// key: 单个字符（或单个英文字母）
// value: 词id集合，所有包含该字符的词
```

**用户输入 "sou" 处理流程：**
1. 拆分每个字符 "s", "o", "u"
2. 分别查每个字符的索引，得到每个字符对应的候选词集合
3. 求**并集**，所有包含任意一个字符的词都是候选
4. 对每个候选词计算和用户输入的**最小编辑距离**
5. 编辑距离越小越相似，按升序排序返回 topN

**最小编辑距离 DP 公式：**
- `dp[i][j]` 表示 `s1[0..i-1]` 和 `s2[0..j-1]` 的最小编辑距离
- 边界：`dp[i][0] = i`，`dp[0][j] = j`
- 如果字符相同：`dp[i][j] = dp[i-1][j-1]`
- 如果不同：`dp[i][j] = min(dp[i-1][j], dp[i][j-1], dp[i-1][j-1]) + 1`
  - `dp[i-1][j] + 1`: 删除
  - `dp[i][j-1] + 1`: 插入
  - `dp[i-1][j-1] + 1`: 替换

### 3.3 Simhash 文档去重

**Simhash 原理（局部敏感哈希）：**
- 目标：相似的文档生成相似的指纹，重复文档海明距离很小
- 算法步骤：
  1. 文档分词，每个词计算权重（TF-IDF）
  2. 每个词计算一个 64bit 哈希值
  3. 对每一位：如果该位是 1，加上权重；如果是 0，减去权重
  4. 最终每位结果 >0 则该位为 1，否则为 0，得到 64bit 指纹
  5. 两篇文章海明距离 ≤ 3 认为是重复

**大规模去重优化：**
- 性质：如果两个 64bit 指纹海明距离 ≤ 3，那么**最高 8 位一定相同**
- 优化：根据最高 8 位**分桶**（2^8 = 256 个桶）
- 只需要和**同一个桶**里的文章比较，不需要和所有文章比较
- 时间复杂度：从 O(N²) → O(N × 平均桶大小)
- N=10000：从 1 亿次比较 → 大约几千次比较，提升几个数量级

### 3.4 并发安全：thread_local Redis 单例

**问题：** hiredis 的 `redisContext` **不是线程安全**的，多个线程并发使用同一个连接会数据竞争

**解决方案：**
```cpp
static thread_local RedisSingleton instance;
```
- 每个线程创建自己独立的 Redis 连接
- 每个线程终身使用这个连接
- 完全不需要加锁，天生线程安全
- 空间换时间，并发性能好

### 3.5 Reactor 模式

**Reactor 组成：**
- **Reactor 线程**：主线程，epoll 监听所有IO事件
- **Acceptor**：接受新连接
- **TcpConnection**：每个连接对应一个
- **EventHandler**：处理可读可写事件
- **线程池**：执行实际业务计算

**工作流程：**
1. 主线程 epoll_wait 等待事件
2. 新连接到来 → accept → 创建 TcpConnection → 添加到 epoll
3. 客户端发送请求 → 可读事件 → 读取HTTP请求 → 打包任务扔进线程池
4. 线程池工作线程取出任务 → 执行搜索计算 → 回发响应

**优点：**
- 分离IO和计算：主线程只处理IO，不做计算
- 线程池利用多核并行计算
- 相比多进程每个进程一个连接，内存占用少，切换开销小

## 四、已完成的优化和修复

根据代码重构总结，已经完成了这些重要修复和优化：

### 4.1 高危 Bug 修复 ✅

| 问题 | 修复方案 |
|------|----------|
| **TF-IDF 计算错误**：原代码硬编码 `N=1, DF=1`，所有词 IDF 都相同 | 修复为正确公式 `IDF = log2(N/(DF+1)) + 1` |
| **多词查询交集错误**：原逻辑只检查第一个词，不保证包含所有词 | 修复为依次计算交集，只保留包含所有词的文档 |
| **Redis 并发不安全**：全局单个 `redisContext`，多线程数据竞争 | 改为 `thread_local`，每个线程一个连接，完全无锁并发安全 |
| **Content-Length 缓冲区溢出**：固定 `char[1024]`，超过溢出 | 改为动态分配 `vector<char>`，增加大小限制 |
| **Redis 内存泄漏**：GET 释放回复，但 SET 后忘记释放 | 添加 `freeReplyObject` 释放 |

### 4.2 性能优化 ✅

| 问题 | 优化方案 | 性能提升 |
|------|----------|----------|
| 停用词每次重新读取文件 | 构造函数预加载一次到 `std::set` 内存 | 巨大提升，避免每次搜索磁盘IO |
| 编辑距离用C99变长数组（栈分配） | 改为 `vector<vector<int>>` 动态堆分配 | 避免长单词栈溢出崩溃 |
| Simhash 去重 O(N²) | 分桶优化，只比较同桶文章 | 从 1亿次 → 几千次比较 |
| docid 线性查找 | 偏移表按顺序存储，直接索引访问 | O(N) → O(1) |

### 4.3 架构改进 ✅

- **分离离线预处理和在线服务**：原来每次启动都预处理要几分钟，现在离线只跑一次，在线几秒钟启动
- **新增双缓存 LRU 一级内存缓存**：热门查询几微秒返回，大大降低 Redis 负载

### 4.4 本次测试新增修复 ✅

| 问题 | 修复方案 |
|------|----------|
| **硬编码绝对路径**：所有路径硬编码为 `/home/wyx/v5/`，项目移动后无法编译运行 | 批量修改所有路径为当前项目正确路径 |
| **getwebPage 栈溢出**：使用 C99 变长数组 `char buffer[offlength + 1]` 在栈上，大文档溢出 | 改为 `std::vector<char>` 动态堆分配，符合标准 C++ |

修复后项目可以正常编译运行，所有严重问题已解决。

## 六、可优化点汇总

### 6.1 算法与数据结构优化

| 优化点 | 现状 | 优化方案 | 预期收益 | 难度 |
|--------|------|----------|----------|------|
| BM25 排序替代 TF-IDF+余弦 | 当前用 TF-IDF + 余弦相似度 | 实现 BM25 排序 | 搜索结果相关性更好，工业界主流 | 中 |
| Trie 树替换前缀索引 | 当前用字符 hash map | 构建 Trie 前缀树 | 更节省空间，前缀匹配更快 | 中 |
| 布隆过滤器快速判断词是否存在 | 没有布隆 | 加一个布隆过滤器 | 不存在的词快速返回，节省时间 | 低 |
| 倒排索引压缩 | 不压缩 | Frame of Reference 压缩 docid | 节省内存，提高 cache 命中率，更快 | 高 |
| 跳表优化交集计算 | set 有序遍历交集 | 添加跳表 | 交集计算更快 | 中高 |

### 阶段四：面试问题准备（持续）

**必须准备的高频问题：**

| 问题 | 要点 |
|------|------|
| 介绍一下你的搜索引擎项目 | 1. 做了什么功能 2. 架构（离线+在线，Reactor，二级缓存）3. 核心算法 4. 你做了什么优化 |
| 什么是倒排索引，为什么用倒排索引？ | 词→文档映射，用户搜索是按词找文档，倒排索引可以快速找到包含该词的所有文档 |
| 说一下 TF-IDF 原理和公式 | TF 是词频，IDF 是逆文档频率，权重 = TF × IDF。IDF = log(N/(DF+1)) + 1。衡量词对文档的重要性 |
| Simhash 原理是什么？怎么优化大规模去重？ | 局部敏感哈希，相似文档得到相似指纹，海明距离≤3认为重复。分桶优化：最高8bit分桶，只比较同桶，O(N²)→近似O(N) |
| 搜索推荐（suggest）是怎么做的？ | 前缀索引找候选，编辑距离排序，返回topN。编辑距离动态规划计算 |
| 说一下最小编辑距离的DP公式 | `dp[i][j] = min(...)` 边界+递推，要能说清楚 |
| 为什么设计成双缓存 LRU？原理是什么？ | 读多写少，双缓存设计：前台读，后台写，定期批量合并，读几乎无锁，减少锁竞争，高并发性能好 |
| LRU 原理是什么？ | 双向链表 + 哈希表，最近访问放表头，容量满了删除表尾，get/put O(1) |
| 为什么用 thread_local Redis 连接？ | hiredis redisContext 不是线程安全，每个线程一个连接，完全无锁并发安全 |
| Reactor 模式是什么？ | 事件驱动，主线程负责IO事件，分发任务给线程池处理计算，分离IO和计算，高并发性能好 |
| epoll 水平触发和边缘触发区别？ | 水平：有数据就一直通知。边缘：只有状态变化通知一次。边缘减少不必要唤醒，性能更好 |
| 项目中哪里用了分治？哪里用了空间换时间？ | 分治：Simhash分桶。空间换时间：前缀索引、倒排索引、缓存 |
| 如果要支持千万级文档，你怎么扩展架构？ | 讨论：分布式索引分割，主从复制，负载均衡，更多缓存层次等 |
